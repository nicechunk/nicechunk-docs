PLAYER, WALLET, AND CONTROL · STEP 10

Skills, XP, and Progress: How One Action Becomes a Level

One accepted action does not update one universal XP bar. Production /play/ first records device-local behavior counters. The Profile screen derives ten browser XP values, reads optional browser-saved XP and selected chain records, keeps the largest value for each skill, and then converts that XP through a browser level curve. Chunk, Game/Smelting, and Backpack/Player separately award four chain-backed XP families under their own program rules. This guide follows /play/index.html → /play/main.js → play/* and the programs those modules call. It explains what refresh or another device can recover, why the three XP sources are never added, and why one Forging XP total can produce two different levels.

Action counter
A device-local number such as minedBlocks, confirmedMines, resourcesCollected, or placedBlocks. It describes browser-observed behavior and is not automatically a public chain record.
Pending
The immediate local stage after the browser predicts an action but before it knows whether the submitted action will be accepted. Pending counters can later be confirmed or reversed.
Derived XP
XP recalculated from the local action counters whenever the Profile skill state is built. It is a formula result, not a separate account or transaction.
Browser-stored XP
An optional localStorage object named for one owner, with an older global fallback. Current production Play reads these objects but does not write them.
Chain XP
XP decoded from a public program-owned account. Precision and Exploration use Chunk Progress, Smelting uses a different Game/Smelting Progress, and Forging uses PlayerProfile.
Effective XP
For one skill, the largest of derived XP, browser-stored XP, and supplied chain XP. The browser chooses a maximum; it does not add the three sources.
Scoped storage
Browser data whose key ends with one wallet owner, so two wallets on the same browser can keep separate local records. Behavior counters use guest when no owner is available.
Legacy fallback
An older unscoped browser key read only when the current owner-scoped key is empty. The two objects are selected with OR, not merged field by field.
Tier XP
The additional XP needed to cross one level step. The Profile panel's Upgrade XP tiers list shows these per-step amounts.
Cumulative threshold
The total XP needed from the beginning to reach a level. It is the sum of all earlier tier amounts and is what level selection compares against.
Progress PDA
A deterministic public account for one owner and GlobalConfig inside one program domain. Identical seed words under different program IDs produce different addresses.
Program domain
The address and ownership namespace created by one Solana program ID. Chunk Progress and Game/Smelting Progress are separate even though both use the words player-progress.
Atomic transaction
All instructions in one Solana transaction succeed together or their state changes are rolled back together. A later Forging XP failure cannot leave the earlier tentative item append behind.
Safe integer boundary
JavaScript Number cannot exactly represent every integer above 9,007,199,254,740,991. A chain u64 can be larger, so converting it directly to Number eventually loses exactness.

Key points

One action can touch several ledgers

A browser pending counter, a confirmed local resource count, and a program-awarded XP increment are different observations. They can use different units and timing without one being a copy of another.

Ten local formulas share seven behavior counters

Mining, confirmed mining, collected resources, placement, confirmed placement, Smelting runs, and processed materials feed ten browser skills with different coefficients. Rollback counters are saved but do not enter these formulas.

Three XP sources compete; they do not accumulate

For each skill the browser takes max(derived, browser-stored, chain). A value of 100 in each source still produces effective XP 100, not 300.

A level step and total threshold are different numbers

The Upgrade XP tiers list shows the cost of each next step. The level calculation sums those costs and compares XP with cumulative totals.

Only four skill families currently receive chain XP

Chunk awards Precision and Exploration, Game/Smelting awards Smelting, and Backpack invokes Player to award Forging. The other six skills currently depend on browser-side data.

Recovery is program-specific

The current Profile refresh can restore Chunk Precision and Exploration plus Profile Forging. Smelting is written on chain, but the refresh reads Chunk's similarly shaped account instead of the Game/Smelting account.

PENDING, CONFIRMED, ACCEPTED

One action can create several progress stories at different moments

Suppose the boy mines one ordinary resource block. As soon as the local mining plan becomes pending, the browser increases minedBlocks. That makes movement-related, Stamina, and Strength formulas react immediately. This is useful feedback, but it is still a device-local prediction and can be reversed.

After the submitted action succeeds, the browser confirms the pending delta, increases confirmedMines, and adds the actual confirmed reward-group item count to resourcesCollected. If submission fails, rollback increases rolledBackMines and removes the planned block count from minedBlocks. rolledBackMines remains useful history, but none of the ten current XP formulas reads it.

The accepting Chunk program keeps a different ledger. For ordinary mining it adds one Precision XP for the accepted block and one Exploration XP only when an extra drop actually exists. Those contract increments are not intended to copy the browser's larger teaching coefficients. A worked example makes the difference visible: if one confirmed mine adds exactly one collected resource item, local Precision rises by 1 × 90 + 1 × 25 = 115 XP, while that ordinary accepted block adds 1 chain Precision XP.

Do not call these numbers synchronized. The browser later compares the local-derived and chain totals and keeps the larger value. It does not add 115 and 1. Also keep four moments separate: pending local prediction, browser confirmation, browser rollback, and program acceptance. One real attempt cannot both confirm and roll back, although an illustration may show an earlier accepted example beside the current pending attempt.

Implementation boundary The same action name does not mean the same unit

A local behavior formula may award tens of derived XP while a program awards one chain XP. The Profile screen's maximum selection hides the smaller total; it does not reconcile or sum the ledgers.

TEN BROWSER FORMULAS

Seven local behavior counters feed ten skills with different weights

The Profile screen derives a complete XP object from seven non-negative local counters: minedBlocks, confirmedMines, resourcesCollected, placedBlocks, confirmedPlacements, smeltingRuns, and materialsSmelted. One action can therefore push several skill cards. Mining contributes to Precision, Swiftness, Exploration, Stamina, Strength, Burden, and Appraisal through different combinations of pending, confirmed, and collected counts.

Placement is deliberately split too. placedBlocks increases while a placement is pending, so Craftsmanship, Swiftness, and Stamina may move before acceptance. Confirmation adds confirmedPlacements, giving Craftsmanship another 85 XP. Rollback restores the Backpack stack, decreases placedBlocks, and records rolledBackPlacements; the rollback counter itself is not used by a skill formula.

Smelting has an important current gap. game-state.js saves and reloads smeltingRuns and materialsSmelted, but those two fields have no production writer and are never incremented by play/play-smelting.js. The formulas would use them; the reviewed success path instead refreshes the authoritative Backpack and interface. A successful chain Smelting action can therefore gain Game/Smelting XP while these browser-derived Smelting, Forging, Burden, Craftsmanship, and Appraisal inputs remain unchanged.

These coefficients are presentation logic, not contract rewards. Precision's 90 and 25 do not tell Chunk how much XP to write. Forging's local material formula does not tell Backpack how to grade an item. The program rules are listed separately in chapter 5.

THREE SOURCES, ONE MAXIMUM

Owner-scoped storage falls back once, then each skill keeps its highest XP source

Behavior counters live under nicechunk.play.profile.v1.<encoded owner>. Without a wallet owner the suffix is guest. Changing wallets makes game-state save the previous owner's profile and load the next owner's profile, so local mining and placement history is not intended to become one shared browser tally.

The optional skill-XP object uses nicechunk.playerSkillXp.<owner>. If that scoped key has no value, the loader tries the older global nicechunk.playerSkillXp key. The older manual-level object follows the same pattern with nicechunk.playerSkills.<owner> and nicechunk.playerSkills. In plain words, it is primary OR legacy fallback, not an object merge. The expression selects one serialized object and never fills missing fields in the primary object from the fallback object.

The Profile state then merges three XP objects by taking the largest non-negative rounded number for each skill: derived behavior XP, browser-stored XP, and the supplied chain XP object. If the values are 300, 900, and 500, effective XP is 900. This protects a larger known total from being replaced by a smaller source, but it is not synchronization, addition, or proof that the largest source is authoritative.

deriveProfileSkillXp always returns all ten skill keys, including zero. profileSkillEffectiveLevel prefers an XP value whenever the key is a finite number. Those ten finite keys—including zero—shadow the legacy playerSkills levels, so the older manual-level object normally does not decide any skill level in this Profile path. A repository search also finds only reads for playerSkillXp and playerSkills in production Play; there is no current localStorage.setItem writer for either family.

Behavior counters nicechunk.play.profile.v1.<encodeURIComponent(owner)>

Uses guest when there is no owner address.

Browser XP nicechunk.playerSkillXp.<owner> OR nicechunk.playerSkillXp

The fallback is used only when the scoped serialized value is absent or empty.

Older manual levels nicechunk.playerSkills.<owner> OR nicechunk.playerSkills

Current production writer none for skill-XP and manual-level keys

The production Play modules read those keys but do not set them.

Implementation boundary Maximum is not authority

The highest visible source wins the browser display even if that source is device-local. To know what a program enforces, inspect the program-owned account and its own threshold table.

FROM XP TO LEVEL

The browser sums rounded tier costs, while its two progress bars answer different questions

Every skill has xpBase and xpGrowth parameters. To price the next step, the browser multiplies xpBase by 10 and by targetLevel raised to xpGrowth, then rounds that one tier to the nearest integer. Reaching a level costs every earlier tier too, so the cumulative threshold is the sum from tier 1 through that level.

Level selection rounds effective XP to a non-negative integer and finds the maximum level whose cumulative threshold has been reached. Upgrade XP tiers prints each per-level incremental cost, not the cumulative total. For example, Forging's Lv.4 tier itself costs 12,865 XP, while its cumulative Lv.4 threshold is 26,628 XP after the earlier three tiers are included.

The detail progress bar removes XP already required for the current level. It divides XP inside the current tier by the next tier cost. After an upgrade, this current amount can return near zero even though total XP never fell. At Lv.10 there is no next tier: current and required become zero, ratio becomes one, and the label switches to Total XP.

The compact skill-card bar is not that tier ratio. It simply shows level divided by 10 and forces at least 4% width so a Lv.0 card still has a visible sliver. Two bars can therefore look different while both match their code. The full cumulative table below includes Lv.1 through Lv.10 for all ten browser skills.

FOUR CHAIN-BACKED XP FAMILIES

Accepted mining, extra drops, Smelting inputs, and Forging grades write different public records

Chunk owns the public Progress used by mining. An ordinary accepted block adds one Precision XP. Range or grouped paths multiply the same per-block constant by the accepted or rewarded block count. Exploration adds one XP for each actual extra drop, not for every eligible roll or every visible resource.

Game/Smelting owns another Progress account with the same seed words but a different program ID. A successful execution first validates selected Backpack inputs, removes them, writes outputs, and then adds one Smelting XP for every selected input index. The multiplier and output count do not replace that input-index rule.

Forging does not use either Progress PDA. Backpack reads Forging XP from PlayerProfile, calculates a final grade, and awards 25 × grade² XP. It tentatively appends the forged record, then invokes the Player program with the calculated outcome. Player verifies the owner-derived current Profile and writes the accumulated u64 at offset 449. If that later invocation fails, Solana rolls the whole transaction back, including the earlier append.

Each Progress PDA is created on demand when the first relevant action needs it. Chunk and Game/Smelting use the same player-progress seed words under different program IDs, so their PDA addresses differ. The current Devnet IDs used by the reviewed website are Chunk GnVKn442KDTDgCyjVG7SEtCQQLjaCiLvrEZDWSU13wbj and unified Game/Smelting 6CurnvneezBuHwPUnrCiFg1QMWeUF67ufQxYebyr2UP7. The Chunk flow's payer can be the action payer or session authority; the Smelting flow uses the owner as payer. Equal seeds therefore do not imply equal addresses, owners, balances, rent payers, or histories.

ONE XP TOTAL, TWO LEVEL TABLES

Forging XP is shared, but the browser and Backpack convert it through different thresholds

Precision, Exploration, and Smelting contract thresholds match their corresponding browser cumulative tables. Forging is different. The Profile browser curve starts Lv.1 at 1,400 XP and grows to a Lv.10 threshold of 242,950 XP. Backpack's fixed contract table starts Lv.1 at 250 XP and reaches Lv.10 at 45,000 XP.

This is not two Forging XP balances. PlayerProfile stores one forging_xp number. The browser converts that total through the browser table for the card. Backpack converts the same total through its contract table when calculating a forged outcome. The resulting level numbers can both be correct for their own consumer.

Three checkpoints make the difference concrete. At 250 XP, Backpack sees contract Lv.1 while the browser remains Lv.0. At 1,875 XP, the browser is Lv.1 and Backpack is Lv.2. At 45,000 XP, Backpack is Lv.10 while the browser is Lv.4; the browser reaches Lv.5 at 45,014 XP. The last pair differs by only 14 XP but crosses a browser threshold.

The dated Devnet snapshot contains owner EDNQ7SfFLR9L51uPdL6CKA6Q7j7PDF84Bsxx9vVELLjx with 1,875 Forging XP, demonstrating the Lv.1 versus Lv.2 conversion against retained account bytes. It is a snapshot at confirmed slot 477,044,477, not proof of a current balance or deployed binary.

REFRESH AND ANOTHER DEVICE

Current refresh restores three skill families correctly and looks for Smelting in the wrong program domain

The Profile chain synchronizer enforces an 18-second cooldown unless refresh is forced. It loads the chain module and requests PlayerProfile, Equipment, Appearance, and one Progress in parallel. The fetchPlayerProgress call derives that Progress PDA under the Chunk program ID and rejects any account not owned by Chunk.

The synchronizer maps positive Chunk precisionGatheringXp, smeltingXp, and explorationXp fields plus positive PlayerProfile forgingXp into the chain XP object; zero chain values are omitted. This is the problem: real Smelting execution writes the same offset shape in the Game/Smelting-owned Progress PDA. fetchPlayerProgress never derives or reads that second address, so a valid Game/Smelting total is absent from the current Profile refresh.

The retained snapshot makes the wrong-domain effect measurable. Owner EDNQ7SfFLR9L51uPdL6CKA6Q7j7PDF84Bsxx9vVELLjx had Chunk Precision 1,678, Chunk Smelting 0, Chunk Exploration 4, Game/Smelting Smelting 29, and Profile Forging 1,875. Current Profile refresh supplies Precision 1,678, Exploration 4, and Forging 1,875, but omits the real Smelting 29 because it reads Chunk's zero-valued field.

On another device, Precision and Exploration can return from Chunk and Forging can return from PlayerProfile. Smelting exists publicly but needs a corrected Game/Smelting fetch before the Profile can recover it. Burden, Craftsmanship, Swiftness, Stamina, Strength, and Appraisal currently depend on device-local counters or optional browser objects. Also note that decoded u64 values are converted to JavaScript Number; above 2^53 − 1, displayed integer precision is not reliable.

For players, use a simple diagnosis order: verify the same wallet is connected, wait for or force a refresh, identify the source and program domain, and never manually add the three browser sources. Source badges would make this easier, but the current Profile interface does not yet show them; that is an improvement proposal, not a description of an existing control.

READ THE IMPLEMENTATION

Fifteen formulas and twenty-two literal excerpts trace counters, stores, thresholds, awards, and recovery

The equations distinguish immediate behavior counters, effective browser XP, tier costs, cumulative thresholds, program awards, and two Level labels. Every source excerpt below is a continuous substring of the reviewed repository. A no-network derived audit reuses the dated account snapshot rather than presenting old bytes as a new live query.

Mining counter lifecycle

pending: minedBlocks += plannedBlocks; confirm: confirmedMines += plannedBlocks and resourcesCollected += confirmedItems; rollback: minedBlocks -= plannedBlocks and rolledBackMines += 1

These are browser counters. Chunk acceptance and its XP award are a separate program result.

plannedBlocks
The number of blocks in this local mining plan.
confirmedItems
The sum of counts in confirmed reward groups.

Placement counter lifecycle

pending: placedBlocks += 1; confirm: confirmedPlacements += 1; rollback: placedBlocks -= 1 and rolledBackPlacements += 1

Rollback also restores the consumed Backpack snapshot. Current skill XP formulas read placedBlocks and confirmedPlacements but not rolledBackPlacements.

placedBlocks
A local count that includes a placement while it is pending.
confirmedPlacements
A separate local count increased only after confirmation.

Ten behavior-derived XP values

Precision=90R+25M; Burden=18R+12T; Smelting=160S+90T; Forging=38T+45S; Craftsmanship=85C+18P+28T; Swiftness=5B+4P; Exploration=44M+10R; Stamina=9B+7P; Strength=12B+6R; Appraisal=22R+36T

B is minedBlocks, M confirmedMines, R resourcesCollected, P placedBlocks, C confirmedPlacements, S smeltingRuns, and T materialsSmelted. All inputs are clamped to non-negative integers.

B, M, R
Mined blocks, confirmed mines, and collected resource items.
P, C
Pending-or-confirmed placed blocks and separately confirmed placements.
S, T
Local Smelting runs and local materialsSmelted; current production Smelting does not write either counter.

Owner-scoped browser selection

StoredObject = JSON(primary owner key OR legacy global key); the primary and legacy objects are not merged

This rule applies separately to browser XP and the older manual-level object. Behavior counters use only their wallet-scoped key and do not use a legacy global profile fallback here.

OR
JavaScript short-circuit selection of one serialized string.

Effective browser XP

EffectiveXP(skill) = max(DerivedXP(skill), BrowserStoredXP(skill), ChainXP(skill))

The merge is performed per key after rounding each value to a non-negative integer. The sources are never summed and chain is not given automatic priority.

max
Choose the greatest value for this one skill.

One browser tier cost

TierXP(targetLevel) = round(xpBase × 10 × targetLevel^xpGrowth)

The Upgrade XP tiers list displays these per-step values. targetLevel runs from 1 through 10.

xpBase, xpGrowth
The two browser parameters stored in each skill definition.

Cumulative browser threshold

TotalXP(level L) = sum of TierXP(k), for k = 1 through L

A tier cost and a cumulative threshold are not interchangeable. Level selection uses the cumulative total.

sum
Add every individual tier cost in the stated range.

Browser level from XP

Level = maximum L in [0,10] where TotalXP(L) ≤ round(max(0, EffectiveXP))

At exactly a threshold, the new level is reached. XP below the Lv.1 threshold remains Lv.0.

L
A candidate browser level from zero through ten.

Progress inside the current browser level

CurrentXP = clamp(XP − TotalXP(currentLevel), 0, TierXP(nextLevel)); Ratio = CurrentXP ÷ TierXP(nextLevel)

At Lv.10, current and required are both zero, ratio is one, and the label shows lifetime Total XP instead.

clamp
Keep a value inside the stated minimum and maximum.

Two different Profile progress bars

CardWidth = max(4%, 100% × level ÷ 10); DetailWidth = 100% × CurrentXP ÷ TierXP(nextLevel)

The visual difference is intentional. The 4% minimum makes a Lv.0 card visible and is not earned progress.

CardWidth
The compact bar shown on every skill card.
DetailWidth
The longer bar inside the selected skill detail.

Chain XP awards

Precision += acceptedBlocks; Exploration += actualExtraDrops; Smelting += selectedInputIndexes; Forging += 25 × finalGrade²

Each quantity is calculated in its accepting program. Precision and Exploration share Chunk Progress, Smelting uses Game/Smelting Progress, and Forging uses PlayerProfile.

finalGrade
The grade after Backpack applies material, item-level, skill, and cap rules; it is between 1 and 10.

Current cross-device recovery mapping

RefreshChainXP = {Precision: Chunk.precision, Exploration: Chunk.exploration, Forging: PlayerProfile.forging}; GameSmelting.smelting is not fetched

The fetched Chunk-shaped object also contains a smelting field, but that field is not the account written by the Smelting execution program.

RefreshChainXP
The positive values supplied to the Profile browser merge after one current refresh.

One Forging XP total, two level functions

BrowserForgingLevel = BrowserCurve(forging_xp); ContractForgingLevel = BackpackTable(forging_xp)

At 45,000 XP the browser level is 4 and the contract level is 10. The browser reaches level 5 at 45,014 XP.

forging_xp
The one u64 total stored in PlayerProfile at offset 449.

The two visible overall Level labels

ProfileLevel = 1 + floor(sum(ten skill levels) ÷ 10); HUDLevel = 1 + floor(confirmedMines ÷ 25)

These labels answer different questions and can disagree without a refresh failure or corrupted account.

floor
Discard the fractional remainder after division.

Exact JavaScript integer boundary

Number(u64) is guaranteed integer-exact only when u64 ≤ 2^53 − 1 = 9,007,199,254,740,991

The current Progress decoder converts readBigUInt64LE results directly to Number. Ordinary observed XP is far smaller, but the type boundary should still be documented honestly.
