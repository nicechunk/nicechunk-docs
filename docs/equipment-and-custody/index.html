PLAYER, WALLET, AND CONTROL · STEP 8

Equipment and Custody: Equipping Selects a Record; Listing Moves It

Equipping a forged item does not move it out of its Backpack. The Player Program validates one record and copies only its 32-byte item_pda into one of nine numeric Profile slots; those slots have no built-in body-part or item-type rules. In the reviewed current path, the later forging-XP call makes every successful output for one owner use that owner's current Profile PDA, so different tools can share the key. The visible and usable tool still depends on a browser hotbar and hash-validated local NCF1 bytes. A Market listing separately snapshots and removes the full 80-byte record. This guide follows production /play/, labels older src/main.js separately, and finishes with dated Devnet evidence.

Solana account owner
The program ID allowed by Solana to change an account's data. It is not the same as the player wallet written inside that data.
Stored player owner
A wallet public key inside Profile or Backpack bytes that a program checks to establish the player relationship.
PlayerProfile
The compact public player account that stores the wallet relationship, one GlobalConfig address, nine equipment references, one selected Backpack reference, and progression fields. It does not contain full tool records or model geometry.
GlobalConfig
A shared configuration account whose address is stored in PlayerProfile and checked by Player, Market, and Chunk flows so unrelated account families cannot be mixed silently.
Custody
The current program-controlled location of the usable 80-byte inventory record: normally a Backpack, temporarily a Listing snapshot, then a destination Backpack after cancel or purchase.
Backpack
A public inventory account with a header and a dense array of block or item records. Equipping reads it; listing can remove one record from it.
Dense record array
A list with no empty gap between active records. Removing one record shifts later records toward the front, so a Backpack index is not permanent identity.
Record index
A zero-based position in the current Backpack array. It differs from a Profile equipment slot, item ID, design hash, and PDA.
PDA (program-derived address)
A public account address deterministically derived from a program ID and seed bytes, with no private signing key. A field named item_pda does not by itself prove that a unique item account exists there.
item_pda
A 32-byte field in a forged Backpack record. Current equip copies it into Profile. Equipment slot
One of nine 32-byte public-key fields in PlayerProfile. The contract enforces only indexes 0 through 8: it assigns no head, hand, or item-category meaning and allows the same key in several slots.
Clear sentinel
Backpack index 255 in the equip instruction. It means write the all-zero public key into a Profile slot and requires no Backpack account.
Forged record
An 80-byte Backpack item record carrying identifiers, item_pda, volume, durability, grade, level, quality, and a 32-bit metadata value used for the design hash.
Design hash
The current 32-bit FNV-1a fingerprint of NCF1 bytes. It helps a browser find matching local presentation data but is not a collision-resistant content identity.
NCF1
The compact forged-object byte format used to reconstruct the visible tool. The current Backpack record keeps only its 32-bit design hash, not the complete NCF1 payload.
Hotbar
The browser's current quick-selection bar. It controls what the local avatar presents and can differ from the nine chain Profile references.
AABB collision
A simple box-to-box contact test whose sides stay aligned with the world axes. Production mining samples tool boxes during a swing and also sweeps between samples so fast motion cannot jump through a block unnoticed.
Local durability
The wear number saved in this wallet's browser hotbar. Current production mining subtracts and can restore this local number, but the submitted Chunk instruction does not change the forged Backpack record's chain durability.
Guardian equipment message
In production Play, a realtime presentation message carrying right-hand kind and variant, a forged flag, design hash, and optional hash-validated NCF1 bytes. It carries neither Profile equipment keys nor Backpack presentation fields and is not a wallet signature or custody certificate.
Rich equipment
Twelve 576-byte records reserved inside PlayerAppearance. Current Appearance packing resets them to empty, so they are not the active nine-slot equip path.
Listing
A 216-byte Market account that can hold a complete 80-byte source record plus seller, price, source, timestamps, state, and buyer history.
Atomic transaction
A Solana transaction whose instructions and program calls all succeed or all roll back. A failed buyer-Backpack append therefore prevents payment settlement in the same purchase transaction.
Local storage
Browser data saved on one device. It can restore NCF1 bytes and hotbar choices, but it can be missing, cleared, or edited and is not chain authority.
Dated snapshot
Account bytes observed through RPC at one confirmed context slot. It can show state at that moment but cannot prove that the reviewed source exactly produced the deployed program binary.

Key points

Equip copies one key and leaves the Backpack unchanged

After wallet, Profile, GlobalConfig, Backpack, and forged-record checks, the Player Program writes only record.item_pda into a numeric slot. It does not remove, lock, reserve, or modify the Backpack record; slot type and duplicate-key rules are not enforced.

One current key can describe several different tools

The audited forge first passes player_profile as item_pda, then the same transaction adds forging XP and forces that Profile to be the owner's current derived PDA. Successful outputs for one owner can therefore share the key while item ID, design hash, durability, grade, level, quality, and Backpack index differ.

The hand on screen is a local reconstruction

Production Play selects from its local hotbar, requires matching hash-validated NCF1 for a usable forged mesh, and checks sampled plus swept tool collision before an impact. Guardian relays presentation only; neither state proves that Profile or Backpack changed.

Market listing is the custody transition

Listing copies the complete 80-byte record into a Listing and removes the dense Backpack record. Equipment source clears the selected Profile slot; Backpack source clears only the first matching slot when the record is an item and a match exists.

Production Play and the older alternate market must not be merged

Production /play/ loads play/main.js and play/play-market.js: create is fixed to Backpack source and cancel omits a destination Backpack. The equipment-source and cancel-fallback code in src/main.js is older or alternate. Neither path exposes set_equipment_slot.

ONE WORD, FOUR DIFFERENT QUESTIONS

Ownership does not tell you where the item record is or what appears in your hand

Players often say I own this tool, but the implementation needs four separate statements. First, a Solana account has a program owner: only that program may directly change its bytes. Second, Profile, Backpack, and Listing data can store a player wallet or seller public key. Third, the complete usable record has a current location. Fourth, the game on one device has a current presentation choice. These statements can agree, but none automatically proves the others.

A Backpack may be owned at the Solana level by Backpack or the unified Game Program while its data names Alice's wallet as the player owner. A forged record inside it can still be Alice's inventory even when no Profile equipment slot refers to it. Conversely, a Profile slot can contain a key while the selected hotbar on Alice's laptop shows something else.

Wallet signatures authorize supported changes; they do not encrypt public account bytes. Program ownership governs writes; it does not mean the program possesses a human wallet. A local mesh proves only that the browser reconstructed and displayed geometry. Use the exact phrase that answers the exact question: account owner, stored owner, record location, or screen presentation.

The current chain equip path accepts Backpack accounts owned by either the standalone Backpack Program or the unified Game Program. Market is build-scoped instead: its standalone build names the standalone Backpack Program, while the unified Game build is compiled with the unified-game feature and names the Game Program. Program domain therefore matters independently of the wallet stored in the Backpack.

SIGNED SELECTION, READ-ONLY INVENTORY

Equipping validates one forged record, then copies only its 32-byte reference

The equip instruction carries two bytes: a Profile equipment slot from 0 through 8 and a Backpack record index. The wallet authority must sign. The Player Program validates the writable current owner-derived Profile, its GlobalConfig relationship, and the slot. For a normal equip it also requires a fourth account: a Backpack owned by the permitted program, with the correct length, magic, version, stored wallet owner, ID-derived PDA, and dense index.

The selected record must be an item in the forged category with nonzero quantity, item ID, and item_pda. Its durability must be positive and not exceed its maximum; grade must be 1 through 10, level 1 through 100, and quality 1 through 10,000 basis points. This reader does not recheck item_code equals the forged constant, volume is nonzero, flags, or the design hash. Those omissions do not make arbitrary data writable—the Backpack account and PDA checks still matter—but they define the exact acceptance boundary.

After those checks, Player writes record.item_pda into one 32-byte Profile field and updates the Profile's shared updated_slot observation field with the current Solana slot. Backpack was borrowed only for reading. Its bytes, item count, record order, durability, and custody remain unchanged. Slots are only numbers 0 through 8: the contract assigns no head, right-hand, armor, or item-category meaning, does not require different keys, and therefore permits the same reference in several slots.

To clear a Profile slot, the instruction uses Backpack index 255 and supplies only authority, Profile, and GlobalConfig. Player writes the default all-zero public key. Clearing a reference does not destroy a tool because the tool's complete record was never stored in that Profile field.

The repository's older TypeScript Player SDK exposes a builder for this instruction. Neither production Play nor the older alternate src/main.js calls that builder or provides another chain set-equipment action. The contract capability should therefore not be described as a complete button workflow already available to ordinary players.

  1. The wallet approves a Profile write

    Player checks the authority signature, owner-derived Profile address, Profile owner and GlobalConfig binding, writable state, and requested slot.

  2. Player reads and verifies one dense Backpack record

    The Backpack remains read-only while its program owner, PDA, stored wallet owner, version, index, forged category, identifiers, and bounded outcome fields are checked.

  3. One public key is copied

    Profile.equipment[slot] becomes Backpack.records[index].item_pda; no Backpack byte is moved, locked, or consumed.

  4. Index 255 clears only the reference

    The Profile slot becomes the zero key without requiring a Backpack account or deleting an inventory record.

What the code guarantees After equip, Backpack_after equals Backpack_before

The instruction has no Backpack mutation. A Profile reference can change, but custody, record bytes, and record order do not.

SHARED REFERENCE, DIFFERENT RECORDS

Current forging can give several different tools the same equipment key

A current verified forge receives canonical NCF1 bytes, checks the design and material requirements, consumes selected material records, calculates the output, and tentatively appends one forged record using the supplied player_profile.key as item_pda. It then invokes Player to add forging XP with that same Profile. Player derives the owner's current Profile PDA and rejects any other address; Solana atomicity rolls back the earlier append if this later call fails. A successful reviewed current forge therefore uses the owner's current Profile key, not an item-specific account.

Two tools forged by the same player can therefore share item_pda while still differing in item ID, Backpack index, design hash, volume, durability, grade, level, or quality. The nine Profile slots retain only the shared 32-byte key. They do not retain those distinguishing fields, so the pointer alone cannot select one unique record when several matches exist.

The full Backpack record is 80 bytes. Its last four bytes hold the design hash, while the complete NCF1 bytes remain in browser storage. Production Play uses the 32-bit FNV-1a hash to find candidate local NCF1, then validates version, length, and bytes against the expected hash before constructing a forged runtime. No persistent item account in this path stores the complete NCF1 payload. On a new device or after storage is cleared, a chain forged reference can still be listed in the interface, but it must not be guessed to have a usable grip or trusted geometry without matching NCF1.

A 32-bit hash has only 4,294,967,296 possible values and is not designed as a cryptographic identity. It is useful as a compact lookup and integrity hint within the current presentation flow, not as proof that two arbitrary designs are globally unique.

The shared key also creates a concrete Market ambiguity. An equipment-source listing scans Backpack records from index zero and selects the first item-kind record with the Profile key. The browser's equipment inventory first builds Map(itemPda to record); repeated keys overwrite earlier entries, so it retains the last matching record. The screen can therefore preview one tool while the contract selects another. A durable fix needs item-unique identity or matching rules that include stable distinguishing data.

PROFILE, APPEARANCE, HOTBAR, AND GUARDIAN

The tool in the avatar's hand is not a direct rendering of the nine chain slots

There are four equipment-shaped structures in the current experience. PlayerProfile has nine 32-byte references. PlayerAppearance reserves twelve much larger rich-equipment records, but current Appearance packing clears all twelve. The browser game state keeps a local hotbar with selected slots and NCF1 presentation data. Guardian sends a realtime equipment message for nearby presentation. These structures have different purposes and should not be merged into one on-chain equipped label.

Production Play equips a Backpack record into its hotbar locally. For a forged record it copies chain metadata, tries to hydrate matching NCF1 from local storage, validates its version, length, and FNV-1a hash, and builds the mesh. The restored geometry and grip determine tool placement and collision parts. Target selection first needs the derived reach sphere to intersect the block, then sampled and swept AABB tests must find an actual swing-path collision. Without matching NCF1, tests require the chain reference to remain non-usable rather than inventing geometry or a grip.

Production play/play-guardian.js sends right-hand kind and variant, a forged flag, design hash, and optional NCF1 bytes only after the payload passes version, size, and hash validation. It does not send Profile equipment keys or Backpack presentation fields. The older alternate src/main.js can add cached Profile keys and Backpack-related fields, but neither packet is a wallet signature, RPC proof, custody transfer, or guarantee that cached chain state is fresh.

When a production swing hits, the browser decreases durability in its wallet-scoped local hotbar and saves that local state. If chain submission is disabled, unavailable, rejected, or throws, the pending mine is rolled back and the local tool damage is restored. Yet play/play-chain-adapter.js merely passes toolSlotIndex to recordBlockBreakOnChain; that chain helper never reads its toolSlot parameter. The Chunk processor checks Profile shape, authority, and GlobalConfig, but not the nine equipment references, and it does not decrement a forged Backpack record's chain durability. Local tool wear and chain item durability are therefore separate today.

Backpack selection has another boundary. The contract can replace Profile.equippedBackpack with another valid Backpack owned by the same wallet, but it defines no all-zero clear sentinel. Production Play does not enumerate and switch among every owned Backpack, and the shared loader tries a wallet-scoped cached Backpack before Profile fallback when the cached account is current, unified-Game-owned, and stores that wallet. Backpack and Chunk reward paths validate the supplied Backpack's ownership but do not require it to equal Profile.equippedBackpack, so the interface should label which source actually won.

  • Profile equipment: nine public-key references written by the Player Program.
  • Appearance rich equipment: twelve reserved records that current packing leaves empty.
  • Local hotbar: the selected browser presentation and interaction source.
  • NCF1 library: device-local complete geometry bytes found by a 32-bit design hash.
  • Production Guardian: held kind and variant, forged flag, design hash, and optional validated NCF1—not Profile keys or custody proof.
  • Mining collision: locally restored geometry must pass reach-sphere plus sampled and swept AABB checks.
  • Durability boundary: local hotbar wear is restored after chain failure; current Chunk mining does not read Profile slots or mutate chain tool durability.
Implementation boundary Visible, locally worn, and chain-equipped are three different claims

A reliable equipment UI should label Profile reference, Backpack record and chain durability, verified local NCF1, selected hotbar and local durability, collision result, and Guardian presentation separately.

FULL-RECORD ESCROW-LIKE STORAGE

Listing, not equipping, moves the complete record out of the Backpack

Creating a current Market listing always receives eight account positions: seller, new Listing, System Program, Backpack, Backpack Program, PlayerProfile, GlobalConfig, and Player Program. It always checks the Player Program ID, but Profile and GlobalConfig data validation is conditional. For direct Backpack source, Market copies and removes the chosen 80-byte record; a block or an item whose item_pda is zero returns no equipment match before Profile validation. A nonzero item enters Profile validation and clears only the first matching slot when one exists. For equipment source, Market validates Profile and GlobalConfig, reads the chosen slot, scans Backpack from index zero for the first item with that key, removes it, and clears exactly the chosen slot.

Removal keeps the Backpack dense: every later record shifts one position toward zero, the old last slot is cleared, and item_count decreases. The Listing remembers the source type and old source index, but it does not store the original Backpack address. That old index is historical context, not stable identity. Ordinary Backpack discard uses the same dense removal primitive but does not write PlayerProfile at all—even the session-authorized account form only reads Profile to validate the session owner. Deleting a referenced record can therefore leave a stale equipment key, and duplicate keys can leave other slots still referring ambiguously to the remaining records.

The Listing is 216 bytes and stores the complete 80-byte source record. During cancellation, current Market requires seller, Listing, a seller-owned destination Backpack, Backpack Program, and Market authority. It appends the saved record to the supplied valid Backpack, clears and closes the Listing data, and returns Listing lamports to the seller. The destination can be any qualifying seller Backpack; it need not be the source account because that address was not retained.

During purchase, the same transaction validates currency accounts, transfers SOL or NCK shares, appends the saved record to the buyer's supplied Backpack through Market authority, and marks the Listing sold with buyer and time. Solana transaction atomicity means a failed Backpack append or later program error rolls back earlier payment changes in that transaction. A sold Listing remains as history while the active item record lives in the buyer Backpack.

Production /play/ loads play/main.js, which imports play/play-market.js. That production market hard-codes new chain listings to Backpack source and its cancel call omits sourceInventory. The create path therefore cannot select an equipment source, and the cancel path cannot form the processor's five-account restore call until it chooses a valid seller destination Backpack.

The repository's src/main.js is an older or alternate path, not the reviewed production /play/ entry. It builds candidates from both Backpack records and Profile equipment slots and supplies listing.sourceInventory, another stored Backpack, or the currently equipped Backpack for cancellation. That destination can still fail ownership, program-domain, or capacity checks and is not guaranteed to be the original source because Listing did not save that address.

The older alternate preview also builds a Map keyed only by item_pda. Because current forge calls can duplicate that key, it retains the last matching record while the Market contract's equipment-source scan chooses the first. Exposing that alternate flow without identity reconciliation can preview one tool and remove another.

  1. Create snapshots then removes

    Market copies the full validated 80-byte source record and removes it. Equipment source clears the selected slot; Backpack source clears only the first matching item slot when one exists.

  2. Cancel appends to a seller destination

    A valid seller Backpack receives the saved record; the Listing is then zeroed, closed, and its lamports returned.

  3. Buy pays and appends atomically

    Payment and buyer-Backpack append share one transaction, so any failure rolls the whole transaction back.

  4. Start with the production /play/ path

    Production play/play-market.js fixes create to Backpack source and omits a cancel destination. Equipment-source creation and fallback cancellation live only in the older alternate src/main.js path.

Implementation boundary A cancel button still needs a valid destination Backpack

Production play/play-market.js currently omits the destination. The older alternate supplies a remembered or current Backpack, but that still requires seller ownership, a supported program domain, and free capacity and does not promise the original source account.

RETAINED CONFIRMED DEVNET OBSERVATION

The live accounts show Backpack records and forged outputs, but no active Profile equipment references

At confirmed context slot 477,030,223 on 18 July 2026, the retained account observation found ten current version-3 Backpack accounts under the unified Game Program, belonging to eight distinct stored owners. Nine were nonempty. Together they held 205 dense records: 200 blocks, three forged items, one material, and one blueprint. Nine Backpacks declared capacity 50 and one historical account declared 51; each held 56,904,960 lamports. The capacity-51 account was created at slot 476,437,186, before the retained current Game deployment slot 476,942,846, so it cannot prove that this deployment still creates capacity 51.

The same context found no current version-5 Listing under the unified Game Program. Of fifteen 773-byte Profile candidates under Player, eleven were canonical owner-derived current version-7 Profiles. Their 99 equipment fields were all the zero public key. Five of those eleven Profiles selected a Backpack; all five addresses existed in the retained Backpack set and all five stored the same owner.

Two named the owner's canonical current version-7 Profile and are consistent with the reviewed successful current forge path. The third named an older noncanonical version-3 Profile; that is historical or deployment-drift evidence, not evidence that today's reviewed path accepts a legacy Profile, because the later Player forging-XP call would reject it and atomically roll back the append.     pub fn write_equipment_slot(
        dst: &mut [u8],
        slot: u8,
        item: &Pubkey,
        updated_slot: u64,
    ) -> ProgramResult {
        if slot as usize >= EQUIPMENT_SLOT_COUNT {
            return Err(NicechunkPlayerError::InvalidEquipmentSlot.into());
        }
        if dst.len() != Self::LEN {
            return Err(NicechunkPlayerError::InvalidPlayerProfileData.into());
        }
        let offset = Self::EQUIPMENT_OFFSET + slot as usize * 32;
        dst[offset..offset + 32].copy_from_slice(item.as_ref());
        dst[Self::UPDATED_SLOT_OFFSET..Self::UPDATED_SLOT_OFFSET + 8]
            .copy_from_slice(&updated_slot.to_le_bytes());

NCF1 equipment bytes must pass length, version, and hash checks

JavaScript play/forge-equipment-payload.js
export const NCF1_EQUIPMENT_VERSION = 14;
export const NCF1_EQUIPMENT_MIN_RAW_BYTES = 14;
export const NCF1_EQUIPMENT_MAX_RAW_BYTES = 640;

export function equipmentPayloadBytes(value) {
  if (value instanceof Uint8Array) return value.length ? value : null;
  if (Array.isArray(value)) return value.length ? Uint8Array.from(value) : null;
  if (value instanceof ArrayBuffer) return value.byteLength ? new Uint8Array(value) : null;
  if (ArrayBuffer.isView(value)) {
    return value.byteLength
      ? new Uint8Array(value.buffer, value.byteOffset, value.byteLength)
      : null;
  }
  return null;
}

export function validatedNcf1EquipmentPayload(value, expectedDesignHash) {
  const bytes = equipmentPayloadBytes(value);
  const designHash = Math.trunc(Number(expectedDesignHash) || 0) >>> 0;
  if (!bytes
    || !designHash
    || bytes.length < NCF1_EQUIPMENT_MIN_RAW_BYTES
    || bytes.length > NCF1_EQUIPMENT_MAX_RAW_BYTES
    || (bytes[0] >> 4) !== NCF1_EQUIPMENT_VERSION
    || forgePayloadHash(bytes) !== designHash) return null;
  return bytes;
}

export function forgePayloadHash(value) {
  const bytes = equipmentPayloadBytes(value);
  if (!bytes) return 0;
  let hash = 0x811c9dc5;
  for (const byte of bytes) {
    hash ^= Number(byte) & 0xff;
    hash = Math.imul(hash, 0x01000193) >>> 0;
  }
  return hash >>> 0;
}

  3. Lines 29–38

    await addTransactionSolSpend(solSpend, conn, initSignature, session.keypair.publicKey);

    pub fn remove_resource_at(
        data: &mut [u8],
        owner: &Pubkey,
        index: u8,
        updated_slot: u64,
    ) -> ProgramResult {
        Self::validate_owner(data, owner)?;
        let item_count = data[Self::ITEM_COUNT_OFFSET];
        if index >= item_count {
            return Err(NicechunkBackpackError::InvalidResourceIndex.into());
        }

        let start = Self::RECORDS_OFFSET + index as usize * BACKPACK_SLOT_RECORD_LEN;
        let end = Self::RECORDS_OFFSET + item_count as usize * BACKPACK_SLOT_RECORD_LEN;
        if start + BACKPACK_SLOT_RECORD_LEN < end {
            data.copy_within(start + BACKPACK_SLOT_RECORD_LEN..end, start);
        }
        let last_start =
            Self::RECORDS_OFFSET + (item_count - 1) as usize * BACKPACK_SLOT_RECORD_LEN;
        data[last_start..last_start + BACKPACK_SLOT_RECORD_LEN].fill(0);
        data[Self::ITEM_COUNT_OFFSET] = item_count.saturating_sub(1);
        data[Self::UPDATED_SLOT_OFFSET..Self::UPDATED_SLOT_OFFSET + 8]
            .copy_from_slice(&updated_slot.to_le_bytes());
        Ok(())
    }

The function shifts later 80-byte records left, zeroes the old tail, decrements item_count, and updates the Backpack. It receives no Profile data, so ordinary discard cannot clear equipment and may leave a stale or ambiguous reference.
