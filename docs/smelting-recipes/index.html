SMELTING 13 · WHOLE RECORDS AND RECIPE AUTHORITY

Smelting Recipe PDAs: Whole Slots, Yield Math, and Authority Boundaries

NiceChunk smelting consumes distinct Backpack records, checks them against the recipe table supplied to the instruction, removes every selected input and fuel record, and appends a bounded set of outputs through the trusted smelting-authority PDA. Recipe tables have compact fixed layouts and deterministic yield math, but the current execution path does not pin a configured table address or table ID. Because table initialization is permissionless and gives the payer authority, the deployed program must not be described as enforcing one canonical public recipe set. Chunk.js supplies efficient material models and heat animation, while local rule data supplies composition and previews; none of those client products are chain authority.

15 min read

Key points

A derived table is not currently a canonical table

Anyone may initialize an unused recipe-table PDA and becomes that table's first authority. Execution checks program ownership and table format, but does not re-derive the address, compare its stored table ID, or require a configured ID. A structurally valid user-controlled table can therefore reach the trusted smelting output append path.

Multiplier means more whole records, not more units inside a record

Each selected Backpack index must be unique, and every recipe input template must match exactly multiplier times through ordered first-fit assignment. Quantity must be nonzero for a valid slot, but its numeric value does not affect recipe matching or partial consumption. Every submitted fuel index is also removed, even when the bundled JSON marks lava heat as consumable false.

The chain computes volume; Chunk.js reconstructs meaning and pixels

Recipe yield, Smelting PlayerProgress, output count, Backpack capacity, and normalized metadata determine the stored result. Temperature labels, burn duration, composition, locally derived attributes, cached material meshes, and the animated furnace core remain client presentation rather than execution proof.

DETERMINISTIC ADDRESS, OPEN CREATION

A table PDA has a precise address, but current execution does not make that address canonical

RecipeTable version 2 uses the literal smelting-recipes seed and a little-endian 64-bit table ID. Initialization verifies that derivation, charges the payer for the fixed account, records the ID and bump, and stores the payer as authority. No administrator or configured-table signer participates. The authority may later upsert a recipe, disable it by replacing the same record with enabled false, or transfer table authority. A separate Civilization receipt adapter can apply a finalized recipe patch to its targeted table.

The critical boundary appears during execution. execute_smelting requires only that the supplied table account is owned by the active Smelting Program, then RecipeTable::find_recipe validates its general format and searches for an enabled recipe ID. The function does not call validate_recipe_table_pda, does not compare the stored table ID or bump with the account address, and does not consult public/mainnet.json or another allowlist. The trusted smelting-authority PDA is still used to append the selected output template into the owner's Backpack.

Together, those facts mean a payer can initialize an unused valid table, become its authority, install structurally valid outputs, and execute them through the same trusted append route. Output item categories are not restricted to the material category, and an output item PDA need only be non-default rather than equal to the table. The present program therefore verifies Backpack records against the supplied table; it does not enforce a canonical recipe set or prove output provenance.

Creation authority Any signing payer

The payer becomes the initial per-table authority after deriving an unused ID.

Execution table gate Program owner and valid bytes

No PDA re-derivation, configured-ID comparison, or whitelist lookup occurs.

Trusted producer reached smelting-authority

The Backpack Program accepts output records signed by this PDA under the Smelting Program.

Implementation boundary Do not call the deployed recipe set canonically enforced

A client allowlist can narrow its own submissions, but the program currently accepts any valid program-owned RecipeTable. Canonical enforcement would require an execution-time PDA or configured-table check, not documentation language.

TEN RECORDS, FIXED TEMPLATES

Each table reserves ten large recipe records, and disabling one does not free its position

The 96-byte header stores magic, version, bump, initialized state, table ID, authority, recipe count, creation and update slots, and creation time. Each 984-byte record then stores recipe ID, enabled flag, minimum heat tier, active input and output counts, yield basis points, eight full 80-byte input templates, four full 80-byte output templates, and an update slot. A complete table is always 9,936 bytes whether it contains one active recipe or ten.

Input count is constrained to one through eight and output count to one through four. Encoding fills unused fixed positions with copies, but only the active counts participate in matching or output. Recipe yield must be one through 10,000. Output item templates require nonzero item identity, PDA, volume, durability, grade, level, and quality fields, yet the format does not restrict a category to smelted materials or tie the item PDA to this table.

Upsert scans all ten records. Reusing a recipe ID replaces its existing position; a new ID uses only a record whose stored recipe ID is zero. Upserting that same ID with enabled false keeps the nonzero ID in its occupied position, but replaces the complete record, including its templates, and writes the current update slot. recipeCount is not reduced. This retained identity is useful for migration history, but a table filled with disabled records is still full and cannot accept another ID without a new table or format change.

Table length 9,936 bytes

A 96-byte header plus ten 984-byte recipe records.

Per-recipe limits 8 inputs and 4 outputs

Inactive fixed template positions remain serialized but are ignored by their counts.

Disabled recipe Not executable, still occupied

find_recipe skips it, while upsert cannot treat its nonzero ID as an empty slot.

How to read this Names and chemistry do not live in RecipeRecord

The chain record carries slot identities, heat tier, yield, and output templates. Human names, temperature in degrees, burn seconds, composition ranges, model shape, and forge-use labels come from client data.

UNIQUE INDEXES, EXACT MULTIPLIER

Smelting consumes complete Backpack records and does not deduct them by quantity

Execution requires the owner wallet itself to sign and be writable. There is no PlayerSession alternative in the Smelting instruction. The supplied Backpack must be owned by the configured Backpack Program, its stored owner must equal the signer, and every selected index must address the current dense record prefix. Input and fuel indexes share one uniqueness bitmap, so one slot cannot serve both roles or appear twice.

The selected input count must equal recipe inputCount times multiplier. The program visits selected indexes in payload order and assigns each record to the first active template that both matches and still needs an occurrence. Every selected block record matches by kind and Packed-Y block ID. Every selected item matches by kind, category, and item code, plus item ID when the template does not use zero as a wildcard. This first-fit rule means order can matter if a wildcard item-ID template overlaps a later specific-ID template. Slot decoding requires BackpackSlotRecord.quantity to be nonzero, but matching does not compare its numeric value, so a quantity-ten record counts once and is destroyed as one whole slot. Smelting also performs no greater-than-or-equal comparison over volume, durability, grade, level, quality, or other material parameters.

Fuel is a separate hardcoded classifier. Current chain tiers recognize dry grass, dead bush, and thorn at tier one; wood-like blocks and material code 1001 at tier two; coal at tier three; and basalt or lava at tier four. Only the maximum submitted fuel tier is compared with minHeatTier. Burn duration and consumable flags are absent. remove_backpack_resources concatenates every input and fuel index, so a selected lava or basalt record is removed even though the bundled JSON declares lava_heat consumable false.

Signer Owner wallet only

The current execute account list has no PlayerProfile or PlayerSession route.

Batch meaning Template repetition

Multiplier repeats every active template across distinct whole records.

Fuel consumption Every supplied fuel index

The highest tier authorizes heat, but all submitted fuel records enter the same removal batch.

Implementation boundary A stack display is not partial-stack protocol support

Quantity remains a field in the 80-byte record, but Smelting has no amount argument and no decrement operation. Select only records the player agrees to destroy completely.

VOLUME IN, BOUNDED RECORDS OUT

Batching increases input volume, while output count remains fixed and failure remains atomic

Before removal, the program computes itemCount minus every selected input and fuel record plus recipe outputCount. If that projected count exceeds Backpack capacity, execution stops. It then removes the dense records through the Backpack Program and appends each active output through the smelting-authority PDA. Backpack append is strict. A later append or progress failure aborts the transaction, so Solana rolls back earlier CPIs and the player does not retain a half-applied state.

Only selected input records contribute actual input volume; fuel does not. A positive stored volume is used directly and zero falls back to 1,000,000 cubic millimetres. For each active output, the program begins with that output's PDA template volume, scales it by multiplier and by the ratio of actual selected input volume to the expected active input-template volume, then applies the recipe yield and Smelting PlayerProgress skill factor. Level zero begins at 10 percent and each level adds five percentage points through level ten, reaching 60 percent. Integer truncation and the positive u32 clamp happen independently for every output; no aggregate pool is divided evenly.

Multiplier does not multiply outputCount. It participates in each output's volume baseline while the larger selected input set supplies the actual volume ratio. Current synchronization writes exactly one output template for each bundled primary and merge material recipe, so one execution appends one output record even for a multi-serving batch. The UI nevertheless displays bundled yieldCount times servings, and several materials have yieldCount two. That number is presentation-only under current synced record construction and can disagree with the confirmed Backpack result.

For item outputs, the program replaces volume and recomputes durability and minimum item level while retaining template identity and quantity. Current synchronization uses the material recipe identity as item ID for each output rather than minting a unique instance ID. The program then adds one Smelting XP per selected input record, excluding fuel. Every one of these mutations belongs to the same atomic instruction.

Capacity projection Remove first, add fixed outputs

The precheck counts records rather than quantities or kilograms.

Batch result More volume, same outputCount

Current primary and merge synchronization each install one output template; that record carries the independently calculated batch volume.

Failure behavior Atomic rollback

Removal, strict append, metadata normalization, and XP update either commit together or do not commit.

What the code guarantees Refresh the Backpack for the authoritative result

A pre-submit preview cannot prove output count, volume, durability, quality, level, or identity. Use the confirmed account bytes after the transaction, especially while client rules and live table records can drift.

BUNDLED RULES VERSUS SUPPLIED TABLE

The current website plans from local rules and does not compare them with the live table

The smelting panel imports src/data/smeltingRules.js through a lightweight adapter. It groups local raw inputs, catalysts, heat tiers, yieldCount, composition, and material identities into recipe cards, caps selection at eight input records, derives a table ID and recipe ID from local material order, and asks for one fuel record. The chain payload itself allows inputCount times multiplier within a combined input-plus-fuel ceiling of ninety-nine, so the current UI is narrower than the Rust instruction.

executeSmeltingOnChain derives the local table ID and checks only that RPC returned nonempty account data. It does not verify account owner in that preliminary read, decode RecipeTable, compare table ID, authority, enabled status, inputs, outputs, yield, or heat, or warn that the local card differs. The wallet therefore signs an instruction whose actual behavior is determined by the supplied on-chain bytes, not the card copy.

Merge planning has another current routing gap. The local matcher can construct a merge candidate from repeated records of one smelted material, but smeltingRecipeChainIdentity always returns the primary material recipe ID and primary 220-series table. It does not branch on recipeKind, so the current primary cards cannot route execution to the installed 320 through 322 merge tables.

CACHED MATERIALS, LOW-COST HEAT

Chunk.js renders the workshop efficiently without turning visual data into authority

Smelted material records store compact identity and metadata, not geometry. Chunk.js maps a recognized material ID to a procedural low-poly builder, emits the shared chunk-object vertex format, collects its palette and finish, and caches both preview meshes and surface profiles with a forty-eight-entry bound. The cyan glass ingot, charcoal pieces, cloth folds, blooms, plates, composites, and dyes are therefore reconstructed client assets rather than bytes embedded in each Backpack record.

The furnace core is a small WebGL2 path, not a general scene. It draws one fullscreen triangle strip with a shader, disables antialiasing, depth, stencil, and buffer preservation, and caps pixel ratio. The play panel updates that visual on a throttled interval while a transaction is pending. Heat tier, progress, pulse, embers, and color communicate state economically, but the animation is not a timer enforced by the program.

The bundled temperatureC and burnSeconds values, chemistry composition ranges, derived attributes, roughness, translucency, emissive color, model description, and forge-use category never enter execute_smelting. The program recognizes fuel through hardcoded block or item identities, reads only minHeatTier and yieldBps from the supplied recipe, and writes normalized slot metadata. Keep UI copy explicit about this boundary so an attractive material preview is not mistaken for verified chemistry or provenance.

Material cache Up to 48 entries per cache

Repeated previews reuse generated geometry and surface profiles instead of rebuilding them per slot.

Core draw One WebGL2 triangle strip

A capped canvas and disabled depth work keep the heat effect small.

Chain authority Recipe bytes and Backpack records

Visual temperature, duration, composition, animation, and mesh fields remain local.

How to read this Performance and verification solve different problems

Caching and a compact shader lower client cost. They do not repair table authorization, validate recipe drift, or prove that a locally previewed attribute was committed on chain.

ADDRESS, RECORD, CAPACITY, AND YIELD EQUATIONS

Five equations expose where records move and where trust is still supplied

The table equation shows a deterministic address and fixed byte budget without implying a whitelist. The matching equation counts unique Backpack records rather than quantity. Capacity and yield equations model the atomic state transition, while metadata scaling explains why the stored output may not equal the local preview.

Recipe, authority, and progress PDAs with fixed table length

P_table(id) = findProgramAddress([UTF8("smelting-recipes"), LE64(id)], smeltingProgramId); P_authority = findProgramAddress([UTF8("smelting-authority")], smeltingProgramId); P_progress(g,o) = findProgramAddress([UTF8("player-progress"), g, o], smeltingProgramId); R = 16 + 8*80 + 4*80 + 8 = 984 bytes; L = 96 + 10*R = 9936 bytes; authority_initial = payer

Initialization validates the table derivation for the requested 64-bit ID, creates the 9,936-byte account, and stores the signing payer as authority. There is no privileged initializer. Execution validates the smelting-authority PDA, but it does not revalidate the supplied table PDA. PlayerProgress is derived from the caller-supplied global-config key g and owner o without pinning g to the canonical Core account.

Whole-record multiplier and ordered first-fit matching

m >= 1; |I| = inputCount*m; allDistinct(concat(I,F)); for each i in payload order, assign i to the first j where matched[j] < m and match(template[j],slot[i]); require matched[j] = m for every 0 <= j < inputCount; match = kind_e == kind_a AND ((kind = block AND blockId_e == blockId_a) OR (kind = item AND category_e == category_a AND code_e == code_a AND (id_e == 0 OR id_e == id_a))); require quantity_i > 0 for slot validity, but quantity_i does not affect matching

I and F contain current Backpack indexes, not stack units. Each selected input record can satisfy only one template occurrence, every selected index must be unique across both arrays, and each recipe template must be matched m times. Assignment follows the input-index payload order and chooses the first active template with remaining capacity that matches, so order can matter when a wildcard item-ID template overlaps a later specific-ID template. Block matching ignores coordinates and other record fields after Packed-Y block-ID recovery. Item identity matching ignores the positive quantity value, flags, coordinates, PDA, volume, durability, grade, level, quality, and metadata; a zero expected item ID is a wildcard.

m
Unsigned batch multiplier carried by the execution payload.
I, F
Selected input-index and fuel-index arrays in the dense Backpack account.
matched[j]
Number of selected whole records assigned to recipe template j.

Capacity-safe whole-slot transition

1 <= |I|, 1 <= |F|, |I|+|F| <= 99; n_after = n_before-|I|-|F|+o; require n_after <= c; removed = I union F; appendedRecords = o

The program checks projected Backpack occupancy before mutation, then sends all input and fuel indexes to dense batch removal and appends exactly the recipe's outputCount records. Current UI selection is narrower: it permits at most eight input records and submits one fuel record. If removal, any strict append, or progress update fails, Solana transaction atomicity rolls the complete instruction back rather than leaving consumed inputs without outputs.

n_before, n_after, c
Occupied records before execution, projected occupied records after execution, and Backpack capacity.
o
Recipe outputCount, constrained to one through four.
99
Combined chain payload ceiling for selected input and fuel indexes; fuelCount must be nonzero.

Per-output PDA volume, input ratio, skill factor, and batch result

V_in=sum(i in I)(storedVolume_i>0?storedVolume_i:1000000); V_recipe=sum(k in activeInputs)(templateVolume_k>0?templateVolume_k:1000000); level=max{k in {0,...,10}|xp_before>=threshold[k]}; skillBps=min(6000,1000+500*level); V_base_j=floor(max(1,V_pda_j)*m*max(1,V_in)/max(1,V_recipe*m)); V_j=max(1,min(4294967295,floor(floor(V_base_j*yieldBps/10000)*skillBps/10000))); chainRecordCount=o, not o*m; current UI displayCount=yieldCount*m

Fuel volume is excluded. Zero stored or template volume uses the one-million-cubic-millimetre default. For each active output j, the program scales that output's own PDA template volume by batch multiplier and by the actual-versus-expected input-volume ratio, then applies recipe and skill factors with integer truncation. It does not calculate one aggregate pool or split that pool evenly. Skill uses XP read before execution, and outputCount remains fixed even when multiplier increases selected inputs. The UI's yieldCount-times-servings display is still not the current chain record count.

V_in, V_recipe, V_pda_j, V_j
Selected input volume, one batch's expected template-input volume, output j's PDA template volume, and the independently calculated volume stored in output record j.
yieldBps, skillBps
Recipe and Smelting PlayerProgress factors, each divided by 10,000.
yieldCount
Bundled UI field used for display; it is not serialized into RecipeRecord.

Output durability and item-level normalization

D_max_j = clamp(roundHalfUp(D_template_j*V_j/1000000),1,u32max); currentBps_j = clamp(floor(max(1,D_current_template_j)*10000/max(1,D_template_j)),1,10000); D_current_j = clamp(roundHalfUp(D_max_j*currentBps_j/10000),1,D_max_j); effective_j = floor(D_current_j*qualityBps_j/10000); level_calc_j = min(100,1+min(80,isqrt(floor(effective_j/25)))+min(20,floor(V_j/500000))); level_out_j = min(100,max(level_template_j,level_calc_j))

For each item output j, the program replaces template volume with that output's independently calculated V_j, scales maximum and current durability, clamps grade to one through ten and quality to one through 10,000, and raises item level to at least the volume-and-durability result. Quantity, category, item code, item ID, item PDA, flags, and metadata otherwise come from the supplied output template. The non-default item PDA is not required to equal the executing recipe table, so it is not cryptographic provenance.

export function createSmeltingMaterialPreviewMesh(options = {}) {
  const materialId = normalizeMaterialId(options.materialId ?? options.id);
  const definition = MODEL_DEFS[materialId];
  if (!definition) return emptyMesh(materialId);
  const cached = modelCache.get(materialId);
  if (cached) return cached;
  const vertices = [];
  const indices = [];
  definition.build(createBuilder(vertices, indices));
  const colors = uniqueColors(vertices);
  const mesh = {
    id: `smelting_material_${materialId}`,
    materialId,
    name: definition.name,
    category: "smelting material",
    className: definition.className,
    shape: definition.shape,
    description: definition.description,
    roughness: definition.roughness,
    translucency: definition.translucency,
    emissive: definition.emissive,
    vertexFormat: "chunk-object",
    vertices,
    indices,
    layers: [],
    colors,
    quadCount: indices.length / 6,
    triangleCount: indices.length / 3,
    vertexCount: vertices.length,
    collision: false,
  };
  modelCache.set(materialId, mesh);
  if (modelCache.size > MODEL_CACHE_LIMIT) modelCache.delete(modelCache.keys().next().value);
  return mesh;
}

The material ID selects a low-poly Chunk.js definition and a cache limited to forty-eight entries. This reconstruction makes inventory and forge previews inexpensive, but no mesh, palette, roughness, translucency, emissive value, or triangle count is committed by the Smelting Program.
