Field Guide / Multi-block Extraction

FOUNDATION 19 · MULTI-BLOCK EXTRACTION

One tree is reconstructed on chain; one collapse graph is not

Tree felling and support collapse can both remove many rendered voxels after one tool action, but they do not carry the same proof. Tree tag 9 receives one trunk coordinate and reconstructs a generated tree inside the Chunk Program. Support collapse has no dedicated instruction: the browser proposes a six-face component, then submits the primary block and each proposed collapse coordinate through ordinary tags 8 and 5. Those coordinates receive canonical mining checks, while the support relationship itself remains a client claim.

18 min read

Key points

Tree membership is program-reconstructed

Tag 9 validates a canonical trunk at the submitted coordinate, derives one generated tree at that x and z, and enumerates its non-air volume from the submitted cut height upward. The client does not supply the accepted tree-block list.

Collapse topology is client-selected

The browser traces six-face components inside a bounded scan, but the chain receives only independent mining instructions. It does not prove adjacency, loss of support, distance from the first block, or membership in one structure.

Atomicity stops at transaction boundaries

Tree records and strict rewards are attempted in one tag 9 transaction. Support collapse commits the primary tag 8 transaction first, then later pairs or single-coordinate retries can leave a confirmed subset that the client must reconcile.

DO NOT MERGE THE CLAIMS

The same visual gesture hides two different authority models

A tool swing can make a tree canopy or unsupported cluster disappear together, but multi-block is only a presentation category. Tree felling has a dedicated, un-namespaced Chunk Program instruction. Its 13 bytes contain tag 9, x, y, z, and the expected trunk block ID. The account list supplies player and Backpack context plus one to four writable ChunkBroken PDAs; it does not supply a coordinate array.

Support collapse has no dedicated program tag or persistent collapse record. The browser computes a proposed component and sends the clicked block first through reward-bearing tag 8. It then submits each proposed collapse coordinate through another tag 8 or ordinary tag 5. The chain can validate each coordinate as a generated-block mine, but no instruction can distinguish those later removals from separately requested ordinary mines.

Tree authority Tag 9 reconstructs membership

One canonical trunk coordinate selects the generated tree and the program derives the affected set.

Collapse authority Client graph, tags 8 and 5

The program receives independent coordinates, not component edges, support anchors, or one collapse identity.

Shared state output ChunkBroken coordinates

Accepted removals are compact per-chunk records; reward-bearing instructions also write the validated Backpack and PlayerProgress.

Implementation boundary Canonical block verification is not structural verification

A support-collapse coordinate can be a real, mineable canonical block and still have no proved relationship to the clicked support. Public descriptions must keep per-coordinate validity separate from topology validity.

TREE TAG 9

The program rebuilds one tree from the cut coordinate

The client tree planner first requires a rendered normal or pine trunk. It walks at most 16 same-trunk cells downward and upward, then collects matching trunk and leaf IDs around that local column, sorts them, and caps its plan at 96 blocks. That plan controls local damage, animation, pending deltas, and the Backpack capacity estimate, but its coordinate list is not serialized into tag 9.

The program validates the player action and Progress/Backpack relationships, independently recalculates the clicked generated block, requires the expected ID to match, and requires that result to be a trunk. It derives the tree candidate at the submitted x and z, requires cut_y to fall inside that trunk, scans y from cut_y through surface plus nine and x/z within two cells, and maps every non-air result to the necessary ChunkBroken accounts. The fixed five-by-five envelope can touch at most four horizontal chunks.

  1. Validate one cut

    Decode a 12-byte payload after tag 9, check canonical bounds, recalculate the clicked block, and require a normal or pine trunk.

  2. Reconstruct one candidate

    Derive the generated tree rooted at the submitted x and z rather than accepting the browser's planned blocks.

  3. Enumerate and route blocks

    Scan the bounded tree volume, derive each chunk and packed coordinate, and match every distinct chunk to one supplied writable PDA.

How to read this Overlap parity is not rechecked per emitted block

The general canonical block query chooses a winner when nearby generated tree volumes overlap. generated_tree_fell_blocks evaluates the selected cut tree directly and does not re-run that global winner comparison for every emitted coordinate, so an overlap-specific equality guarantee is not currently covered by a test.

TREE REWARDS AND CURRENT DEFECTS

One atomic tree transaction still has cut-height, duplicate, and foundation gaps

Within one successful tag 9 execution, every reconstructed coordinate is appended, every trunk earns a strict Backpack resource, every fifth enumerated leaf earns another strict resource, and Precision Gathering XP increases by the same reward count. Any failed strict append, ChunkBroken growth, account check, or later transaction instruction rolls back the whole Solana transaction. The tree path does not read ResourceDropTable or SurfaceDecorationTable, so it has no exploration-extra or face-decoration reward branch.

Atomicity does not correct the set disagreement. The local planner begins at its discovered trunk base, while tag 9 begins at the clicked cut_y. If the player strikes a higher trunk block, the local pending plan includes lower trunk cells that the program did not record. The returned tree result contains no confirmedBlocks list, so the support-collapse intersection logic cannot narrow that tree plan before local confirmation.

Duplicate protection is also incomplete. The instruction checks contains_packed only for the clicked coordinate, then directly appends every other reconstructed coordinate. After a high cut, a later cut on an unbroken lower trunk can revisit upper coordinates, append duplicate ChunkBroken records, and select those blocks for rewards again. Finally, tag 9 receives no FoundationChunk account and never calls the foundation-protection predicate used by tags 5 and 8.

  • The client preflights Backpack capacity from its local reward estimate, while the program settles rewards from its independently reconstructed list; disagreement can turn into an atomic transaction failure.
  • ChunkBroken append_packed enforces header validity and capacity but performs no duplicate search itself.
  • A duplicate record consumes another three-byte slot and can repeat the corresponding strict tree reward under the lower-cut sequence.
  • No FoundationChunk, ResourceDropTable, or SurfaceDecorationTable account appears in tag 9's fixed account prefix.
  • The current client sends one tree transaction with a 1,400,000 compute-unit limit and has no tree-specific split or retry path.
Implementation boundary Current tag 9 is not duplicate-safe or foundation-aware

These are active audit findings, not hypothetical properties already fixed elsewhere. A verifier should not treat tree reconstruction alone as proof that every emitted coordinate was previously unmined or outside a protected foundation.

CLIENT SUPPORT MODEL

The browser iteratively removes unanchored six-face components

The planner accepts solid, mineable, non-fluid blocks other than air and bedrock. Around the primary coordinate it opens a box extending seven cells in x and z, eight downward, and twenty upward. It starts only from face neighbors of coordinates already planned for removal, traces each component through the six axis-aligned neighbors, and treats minimum-world height, adjacent bedrock, or escape from the scan box as support.

When a component has no anchor, its cells join the planned-removed set. The planner repeats because removing one unsupported component can expose another. It stops after 48 collapse blocks. For a non-empty result, a coordinate-derived score orders reward candidates and keeps at least one or floor thirty percent, whichever is larger. All of these limits, edges, anchors, iterations, and scores exist in JavaScript only.

Neighborhood 6 face directions

Diagonal and corner contact do not join components in the current planner.

Scan bounds ±7 x/z · -8/+20 y

Touching the boundary is treated as evidence of support rather than permission to continue scanning.

Client proposal cap 48 collapse blocks

The cap is enforced before submission; no Chunk Program collapse state carries this number.

Implementation boundary A modified client can propose an unrelated coordinate list

The standard adapter deduplicates and canonically resolves up to 48 proposed coordinates, but neither tag 5 nor tag 8 receives the original support, a graph edge, reach bound, or anchor result. Program acceptance therefore does not prove that the standard planner produced the list.

WHAT THE CHAIN DOES CHECK

Every proposed collapse block is still a canonical mining request

Before building transactions, the standard adapter truncates the proposal to 48 unique coordinates, excludes the primary key, recalculates their canonical block IDs through the client protocol mirror, and filters unmineable results. This is useful preflight, not authority. Each resulting tag 5 or tag 8 instruction supplies x, y, z and expectedBlockId, and the Chunk Program independently performs its own generated-block calculation.

Both tags validate the active player session, GlobalConfig relationship, target ChunkBroken PDA, target FoundationChunk PDA, canonical expected block, mineability, and prior broken state. Tag 8 additionally validates PlayerProgress, both rule-table PDAs, the Backpack Program and Backpack, appends the required base resource, attempts up to two secondary rewards lossily, and advances progress. These checks are strong per coordinate and silent about the relationship among coordinates.

  • The primary clicked block always uses reward-bearing tag 8 and must succeed before collapse batches begin.
  • Selected collapse reward candidates use tag 8; other confirmed collapse coordinates use non-reward tag 5.
  • Foundation protection applies to every tag 5 and tag 8 coordinate through its own target chunk account.
  • The program does not verify face adjacency, unsupported status, distance from the primary, common material, one component, client scan bounds, or a 48-block sequence cap.
  • A direct caller can submit ordinary canonical mining instructions without calling the standard support planner; the chain has no collapse identity by which to reject that distinction.
What the code guarantees Per-coordinate verification remains meaningful

A confirmed collapse coordinate is not an arbitrary client block ID. It passed the same canonical generated-block, duplicate, session, and foundation gates as its underlying tag 5 or tag 8 instruction. What remains unproved is why those individually valid mines belong together.

PAIR BATCHES AND RECONCILIATION

The primary can commit even when every later collapse block fails

The adapter loads the authoritative Backpack, reserves the primary slot, chooses the reward-bearing collapse subset, and initializes the affected ChunkBroken accounts. It then signs and confirms the primary tag 8 transaction by itself. Only after that success does it partition collapse coordinates into batches of two, with each later transaction carrying one or two independent mining instructions.

When a pair fails, the helper retries each coordinate in its own transaction. One recovered single allows later batches to continue. If both singles fail, every later block is marked aborted and no more batches are submitted. The helper does not distinguish shared infrastructure failure from two unrelated coordinate rejections before making that decision.

Each individual Solana transaction is atomic, but the overall sequence is not. If the primary succeeds and every collapse transaction fails, the function still returns submitted true, the primary signature, confirmedBlocks containing the primary, and partialCollapse true. Successful earlier collapse transactions also remain committed. The local result reconciler intersects the pending proposal with confirmedBlocks so failed and aborted voxels are not finalized as part of that local action.

  1. Commit the primary

    Send one reward-bearing tag 8 transaction and stop the whole call if it throws.

  2. Submit pairs

    Put at most two canonical mining instructions in each later transaction, preserving atomicity only within that pair.

  3. Retry and possibly abort

    Split a failed pair into singles; if neither recovers, mark every remaining proposal as batch-wide-failure.

  4. Commit the intersection

    Return all confirmed signatures and coordinates, then reduce pending blocks, air deltas, collapse blocks, and reward groups to that subset.

How to read this Partial success is a first-class result

submitted true means the primary transaction committed, not that the whole planned collapse committed. Consumers must inspect partialCollapse, confirmedBlocks, failedCollapseBlocks, and the signatures array.

SETS, REWARDS, AND COMMIT BOUNDARIES

Six equations expose what is reconstructed, proposed, rewarded, and retained

The tree equations describe tag 9's generated set and strict reward count. The support equations describe the browser's graph rule, its conservative reward selection, and the exact confirmed-set intersection used after a partial sequence.

Tag 9 reconstructs a bounded generated-tree set

F_9(c) = {p | cutY <= p_y <= surface(c_x,c_z)+9 AND |p_x-c_x|<=2 AND |p_z-c_z|<=2 AND V(tree(c_x,c_z),p)!=air}; envelope(F_9)<=5*5*9=225

After the clicked coordinate c is independently recalculated as the expected trunk, the program derives the generated tree rooted at c_x and c_z and scans this fixed envelope. Only non-air results from that one tree volume enter F_9. The actual set is smaller than the 225-cell envelope and may span at most four 16-by-16 ChunkBroken accounts.

c
The submitted integer cut coordinate, including cutY.
V(tree,p)
The Chunk Program's generated volume result for the selected tree at coordinate p.
F_9
The block list reconstructed by tag 9; it is not decoded from a client-supplied coordinate array.

Tree rewards count every trunk and each fifth enumerated leaf

R_9 = T + floor(L/5); XP_precision = R_9

T and L are counted in the program's y, z, x enumeration order over F_9. Each selected reward uses a strict Backpack append, and the current Precision Gathering increment is one XP per selected reward. ResourceDropTable and SurfaceDecorationTable do not participate in tag 9.

T
The number of reconstructed normal or pine trunk blocks.
L
The number of reconstructed normal or pine leaf blocks.
R_9
The number of strict base-resource appends attempted by the tree instruction.

The current client and tag 9 start at different heights

minY(F_client)=baseY_local; minY(F_9)=cutY; cutY>baseY_local => F_client-F_9 contains lower trunk coordinates

The browser walks down from the hit to the local trunk base before planning its full visual removal. Tag 9 begins its scan at the submitted hit height. A non-base strike therefore has a predictable lower-trunk disagreement, and the current tree result does not return a confirmed coordinate list that could trim the local plan.

baseY_local
The lowest same-trunk block found by the client's bounded downward scan.
cutY
The y coordinate encoded in the 13-byte tag 9 instruction.
F_client-F_9
Blocks the local plan can remove visually even though that tag 9 execution did not record them.

The support decision is a local six-face graph predicate

collapse(C) = connected_6(C) AND NOT(reachesScanBoundary(C) OR reachesMinY(C) OR touchesBedrock(C))

The planner treats each mineable solid voxel as a vertex and face contact as an edge. Reaching outside the bounded scan, the configured minimum world height, or a bedrock neighbor makes the component supported. This predicate is never encoded into a Chunk Program instruction, so it is not an on-chain invariant.

C
A component discovered next to a block already planned for removal.
connected_6
Connectivity through positive and negative x, y, and z face neighbors only.
reachesScanBoundary
The trace encounters a neighbor outside the local radius and vertical reach, which the planner treats as support.

Collapse reward candidates are capped twice

q(N)=max(1,floor(3N/10)); r(N,S)<=min(q(N),floor((S-1)/3)), for N>0

The planner deterministically selects q candidates from N collapse blocks. Before submission, the chain adapter reserves one Backpack slot for the primary block and conservatively budgets three slots for every selected reward-bearing collapse instruction: its required base block plus as many as two lossy secondary drops. The final number r can therefore be smaller than q.

N
The number of client-planned collapse blocks, excluding the primary block.
S
Available Backpack record slots observed before the sequence begins.
r
The collapse coordinates selected for reward-bearing tag 8 rather than non-reward tag 5.

Local commitment is reduced to the confirmed coordinate subset

K_commit = K_pending intersection ({primary} union K_confirmed_batches)

The primary tag 8 transaction is sent first. The adapter reports only later blocks whose pair or retry transaction succeeded, and reconciliation intersects that returned set with the original pending proposal. Failed and aborted coordinates are removed from pending air deltas and reward groups; already-confirmed transactions cannot be rolled back as one larger unit.

K_pending
Coordinate keys in the original browser mining proposal.
K_confirmed_batches
Collapse coordinate keys from successful pair or single-coordinate transactions.
K_commit
The subset retained when the local pending action is confirmed.
