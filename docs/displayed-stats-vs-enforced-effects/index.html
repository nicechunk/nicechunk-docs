PLAYER, WALLET, AND CONTROL · STEP 9

Displayed Stats vs Enforced Effects: A Number Needs a Consumer

NiceChunk can store a number, calculate a card from it, and describe a future effect without using that number in gameplay. Current production /play/ directly applies Swiftness to local movement. Chunk independently recalculates Precision Gathering and Exploration, Smelting independently recalculates output, and Backpack independently recalculates Forging. Burden, Craftsmanship, Stamina, Strength, and Appraisal have no current production consumer. The six base Profile attributes also remain fixed defaults with no survival, combat, carrying, mining-power, or building-power loop. This guide follows /play/index.html → /play/main.js → play/*, keeps older src/main.js out of the production claim, and checks those source boundaries against one dated confirmed Devnet snapshot.

Stored field
A value written into account or browser data. Storage proves the value exists there; it does not prove that any action reads it.
Displayed value
Text or a card calculated for the interface. It may be useful and accurate as a display while still having no effect on a transaction, contract result, or game rule.
Consumer
The specific production code that reads a value and uses it to change an outcome such as movement distance, recovered volume, an extra-drop chance, Smelting output, or forged durability.
Browser-local effect
A real effect inside this running browser, such as Swiftness changing the local movement controller. It is not automatically checked or enforced by Guardian or a Solana program.
Contract-enforced effect
A program reads authoritative account data and calculates the accepted result itself instead of trusting a number supplied by the browser.
Derived value
A value calculated from other records or counters. The Profile screen's six stat cards, two different Level labels, and Reputation are current examples.
Effective skill XP
For each skill, the browser takes the largest XP value among local behavior counters, browser-stored skill XP, and the chain XP object, then converts that XP through the browser threshold table.
Legacy level object
The older nicechunk.playerSkills.<wallet> browser object. It can remain as a fallback, but the normal XP path supplies every skill key—even zero—so the XP-derived level usually wins.
PlayerProfile base attributes
Six u16 fields named health, energy, stamina, miningPower, buildPower, and defense. They are distinct from the ten browser skill cards.
Progress PDA
A program-derived public account that stores skill progress for one owner and GlobalConfig inside one program domain. The same seed words under different program IDs create different addresses.
Basis points (bps)
A scale where 10,000 means 100%. Contracts use integer basis points so they do not need floating-point percentages.
Pending prediction
The browser's immediate estimate before chain acceptance. It helps presentation, but the submitted instruction and program calculation decide the authoritative result.
Guardian
The realtime regional service that relays pose, identity, equipment presentation, chat, and dig hints. Its current movement protocol does not carry Profile attributes, skill levels, or XP.
Dated snapshot
Account bytes observed through RPC at one confirmed context slot. It proves what those accounts contained then, not a permanent population total or reproducible deployed binary.

Key points

A label is not a rule

Ask which production consumer reads the value and what accepted result changes. A field can be stored forever and a card can update perfectly while neither affects movement, mining, carrying, crafting, combat, or settlement.

The six Profile defaults are not a survival system

health 100, energy 100, stamina 100, miningPower 1, buildPower 1, and defense 0 are initialized on chain, but current Player instructions do not edit them and production /play/ has no gameplay consumer for them.

Only one browser skill directly changes ordinary local control

Swiftness multiplies the base movement speed used by the local controller. Guardian and the Player Program do not read its level or enforce a skill-limited speed, so this is real browser behavior rather than chain-enforced racing state.

Mining, Exploration, Smelting, and Forging are recalculated by programs

The browser may preview a result, but Chunk reads Chunk progress, Smelting reads Smelting progress, and Backpack reads PlayerProfile forging_xp before applying its own thresholds and rules. Precision, Exploration, Smelting, and Forging therefore have contract consumers, with important UI-versus-contract differences explained below.

One seed name can hide two independent skill histories

Chunk and Game/Smelting both derive [player-progress, GlobalConfig, owner], but their program IDs differ. Current Profile fetching reads the Chunk account while real Smelting uses the Game/Smelting account.

Current means current

An unconsumed field may become gameplay later. This page records the reviewed production path and a dated Devnet snapshot; it does not promise that a future release will keep these boundaries.

STORAGE, DISPLAY, EXECUTION

A number becomes a rule only when an active consumer uses it

Imagine a character sheet that says Strength 20. Three different things may be true. The number may be saved. The interface may print it. A lifting, tool, damage, or carrying system may read it. Only the third statement tells us that Strength changes gameplay. NiceChunk currently contains examples of all three stages, so the word active is too vague unless we name the consumer.

Use a four-question test. Where did the value come from? Which production function reads it? Does that function change a visible action, a submitted instruction, or a program result? Can another authority recalculate the result without trusting the browser? This test avoids two opposite mistakes: calling every card fake, and calling every displayed formula an enforced rule.

Swiftness demonstrates a real browser-local consumer: the production main module writes a multiplier into controls.speed and movement uses that speed. Precision demonstrates a preview plus an independent contract consumer: the browser estimates pending volume, does not submit that estimate, and Chunk recalculates volume from its own progress account. Burden demonstrates display without a consumer: its carry kilograms appear on the Profile screen, but no current inventory weight, overweight, or movement rule reads them.

The production boundary matters. /play/index.html loads /play/main.js, which imports the play/* modules reviewed here. The older /src/main.js contains related historical or alternate logic but is not imported by this /play/ entry, so it cannot be used to prove a current consumer.

Consumed A production outcome changes

Recalculated The accepting program derives its own result

Implementation boundary Never stop at the card

When a guide says a stat works, it should be able to finish this sentence: production code reads this exact value here, and it changes this exact outcome.

SIX STORED DEFAULTS

health, energy, stamina, miningPower, buildPower, and defense are reserved state—not current gameplay loops

PlayerProfile initialization writes six values in a fixed order: health 100, energy 100, stamina 100, miningPower 1, buildPower 1, and defense 0. They are real public account bytes. They are not the six cards shown in the current Profile overview, and the creation screen does not allocate different attributes to the boy and girl models.

The current Player instruction router can initialize a player, update position, select equipment, change Backpack style or pointer, manage a session, add forging XP, change name, write Appearance, close Appearance, and maintain invite pages. After initialization or migration, it has no updater for these six base attributes. A repository-wide review of the production /play/ path also finds no action that reads them to decide damage, fatigue, speed, mining yield, building strength, carrying, or defense.

The two Stamina names are especially easy to merge by accident. Profile.stamina is one chain field initialized to 100. Stamina Lv.0–10 is one browser skill whose effect object calculates a fatigue multiplier. The current code does not connect the field to the skill, and no fatigue consumer uses either value.

The retained Devnet snapshot queried all three relevant programs at confirmed context slot 477,044,477. Fifteen 773-byte Profile candidates were returned; eleven were canonical current version-7 Profile PDAs. All eleven stored exactly the six defaults, and zero canonical Profile had a nondefault base attribute. That is evidence about those accounts on 18 July 2026, not proof that every historical or future Profile must match.

How to read this Reserved does not mean useless

Keeping stable account fields can support future systems. The safe present-tense statement is that the reviewed production path does not consume them today.

PROFILE SCREEN SOURCES

The six visible stat cards, two Level labels, and Reputation are browser-derived

The Profile overview does not print the six base Profile attributes. It converts effective browser skill levels into Gathering, Carry, Smelting, Speed, Strength, and Crafting cards. A card can therefore change when browser XP changes even though health, energy, Profile stamina, miningPower, buildPower, and defense remain untouched on chain.

For each of the ten skills, buildProfileSkillState collects XP derived from local action counters, XP saved under nicechunk.playerSkillXp.<wallet>, and the chain XP object supplied to the screen. mergeSkillXp keeps the largest value for each skill. It does not add the sources together. The older nicechunk.playerSkills.<wallet> level object is still loaded, with an unscoped nicechunk.playerSkills fallback, but deriveProfileSkillXp supplies every skill key—even zero—so profileSkillEffectiveLevel normally uses the XP-derived branch instead of that legacy manual level.

The word Level also labels two independent calculations. The small top HUD uses one plus local confirmed mines divided by 25 and rounded down. The Profile overview uses one plus the sum of ten effective skill levels divided by ten and rounded down. They can disagree without either value being corrupt. Reputation is another local formula: confirmed mines times five, confirmed placements times three, plus materials smelted times four.

These values can still be useful progress summaries. The important wording is browser-derived, not PlayerProfile attributes and not automatically chain-enforced. If browser storage is cleared, changed, or restored on another device, locally derived portions can differ until authoritative reads and new actions rebuild them.

TEN SKILLS, TEN QUESTIONS

The consumer matrix shows exactly which skill effects cross into production behavior

createProfileSkillEffects returns ten convenient fields. Returning a field is not the same as consuming it. The review follows each field through production /play/, Guardian, and the relevant programs, then labels the strongest claim supported by that path.

Four skills have program-side consumers: Chunk executes Precision Gathering and Exploration; Smelting executes Smelting; Backpack executes Forging. One skill has a direct ordinary browser-control consumer: Swiftness. Five fields—Burden, Craftsmanship, Stamina, Strength, and Appraisal—currently stop at calculation or display.

The four contract-backed skills are not all correctly represented by the current Profile screen. Precision is a local preview followed by contract recomputation. The reviewed Smelting source path derives its separate progress account, while the current Profile fetch reads the Chunk progress account instead. Forging's browser threshold and simple +5% description disagree with the Backpack contract. Exploration's browser effect field is unused even though Chunk reads Exploration XP directly.

This matrix is deliberately conservative. A code comment, tooltip, planned stat, or similarly named field does not count as a consumer. The result must reach movement, accepted inventory output, accepted drop probability, accepted forged outcome, or another concrete production behavior.

PRECISION AND EXPLORATION

The browser estimates mining; Chunk weighs the accepted reward again

After the third valid physical tool contact, the mining controller calculates a pending yield from precisionGatheringBps. That estimate helps build local reward groups and an immediate presentation. The chain adapter's intermediate blockFromPending object keeps integer coordinates, blockId, resourceId, and a local key but omits gatheringYieldBps and resourceVolumeMilliLiters. The later ordinary Chunk instruction encodes only x, y, z, and expectedBlockId from that path.

Chunk validates or creates its own player-progress PDA, reads precision_gathering_xp, converts it through the contract threshold table, and calculates recovered volume from the canonical 1,000,000 mm³ resource block. The program then writes that calculated volume into Backpack output. Editing a browser preview therefore cannot directly instruct Chunk to credit a larger volume.

Exploration follows the same authority principle but changes a different result. Chunk reads exploration_xp from the same Chunk progress account, converts it to a level, and multiplies an eligible extra-drop rule's chance by 100% plus 10% per level, capped at 10,000 bps. The browser's rareRollWeightBps field is not what the instruction submits.

Precision and Exploration are real contract effects, but they do not make every mining input authoritative. The browser still controls the local swing presentation, pending state, and tool collision sampling; Guardian supplies realtime hints; the Chunk instruction and program checks decide accepted block and reward changes. Strength, miningPower, Profile stamina, and the Stamina skill do not change the current three-contact rule.

TWO PROGRESS DOMAINS, TWO FORMULAS

Smelting uses a different Progress PDA, and Forging ignores the browser's simple bonus

Chunk and Game/Smelting both use the seed tuple ["player-progress", GlobalConfig, owner]. A PDA also includes the deriving program ID. Chunk's address under GnVKn…13wbj and Smelting's address under 6Curn…2UP7 are therefore different accounts even for the same wallet and GlobalConfig.

Current fetchPlayerProgress derives through context.chunkProgramId and accepts only an account owned by that Chunk program. It exposes the smeltingXp bytes found in that Chunk-format account. The reviewed createExecuteSmeltingInstruction source instead calls deriveSmeltingPlayerProgressPdaForContext, derives under context.smeltingProgramId, and supplies that separate account to Smelting. The Profile screen can therefore feed the wrong Smelting XP into its effective-XP calculation without changing which progress PDA this instruction builder supplies.

Smelting reads its own progress.smelting_xp, converts it to skill_output_bps, checks recipe inputs, removes resources, and calculates output from declared recipe volume, actual input volume, batch size, recipe yield, and skill output. The browser also calculates skillOutputBps inside selectedRecipeView, but that value is only returned in the local view: it is neither displayed in the current progress status nor included in the Smelting instruction.

Forging has a second, larger mismatch. The browser skill definition uses xpBase 140 with a global multiplier of 10, so browser Forging Lv.1 starts at 1,400 XP, and the card describes +5% durability per level. Backpack's contract table starts Lv.1 at 250 XP. Its outcome combines material volume, effective durability accumulated from current durability times quality, maximum durability, weighted grade, a weak-material cap, item level, skill grade and cap, a candidate factor, and a material durability cap. The simple browser bonus field is not a contract input.

The confirmed Devnet snapshot found four canonical Chunk progress accounts and nine canonical Game/Smelting progress accounts. Four owners had both. For each of those owners the seed text and GlobalConfig relationship matched, but the two program IDs produced different addresses and independent XP histories.

REAL LOCAL MOVEMENT

Swiftness really changes local walking speed, but no authority enforces a skill-limited race

Production /play/ defines a base speed of 14.8. When Profile skill effects refresh, movementSpeedMultiplier becomes (100 + 3 × Swiftness level) / 100 and the main module writes base speed times that multiplier into controls.speed. The Chunk.js control update then uses controls.speed to calculate this frame's horizontal movement. This is a concrete production consumer, not merely a card.

The boundary is equally important. Holding Shift multiplies the current control speed by sprintMultiplier, which is configured as five. Flight uses the same current speed for horizontal and vertical motion. None of these paths subtract health, energy, Profile stamina, or the Stamina skill's fatigue multiplier.

The Player position instruction checks signature, Profile relationships, and the world's Y bounds before saving a position; it does not receive Swiftness XP or verify distance divided by time. Guardian's MOVE handler decodes a pose, applies a message-rate limit, maps local to global chunks, enforces Guardian service range, and updates the pose. It likewise receives no skill level and performs no skill-speed check.

So Swiftness works for ordinary local control, but it is not an anti-cheat certificate, competitive racing attribute, or proof that a remote observer moved legally. A future race or authoritative locomotion system would need its own timing, path, collision, session, and skill checks.

WHAT IS NOT IN THE LOOP

Guardian carries pose and presentation, not a hidden combat, weight, or fatigue system

The browser's MOVE packet contains local chunk coordinates, compact offsets, Y, yaw, pitch, and a client tick. The equipment packet contains a sequence, right-hand kind and variant, flags, a design hash, and optional payload bytes. Player identity carries a name. None of these packets carries health, energy, Profile stamina, miningPower, buildPower, defense, ten skill levels, or skill XP.

Guardian checks packet shape, connection state, rate limits, sequence behavior, service range, and the fields needed for realtime pose or presentation. Its equipment message helps remote clients reconstruct a right-hand item; it is not a wallet signature or proof of Profile attributes. Its dig path validates regional protocol rules and publishes hints; accepted resource settlement still belongs to the relevant Solana program.

Current production /play/ has no enemy, attack, incoming damage, healing, death, armor, or defense loop. Combat appears as an inventory category, not a functioning stat consumer. It also has no aggregate inventory mass, overweight state, fatigue meter, action-energy cost, Strength gate, Appraisal information gate, or Craftsmanship recipe tier. Mining's ordinary target still resolves after three valid physical tool contacts.

This absence should be documented without pretending the fields are permanently meaningless. A future feature can connect a stored field by adding a consumer and defining who owns the result. Until then, player-facing copy should label the five unconsumed skill effects as preview, planned, or informational and should not promise survival, carrying, crafting, or combat benefits.

The dated snapshot strengthens only the stored-state claim: all eleven canonical version-7 Profiles at confirmed slot 477,044,477 retained the six defaults, and the two progress families were observably separate. It does not prove that the deployed program binaries match this dirty working tree byte for byte, so live account evidence and source-consumer evidence remain two distinct truth layers.

Current combat No gameplay loop

No enemy damage, HP reduction, healing, death, armor, or defense settlement.

No total kg, safe-capacity gate, overweight state, or Burden slowdown.

To activate one of these fields safely, define the authoritative source, the action that reads it, the exact formula, the signer or session rules, the accepted result, and how the interface distinguishes preview from settlement.

READ THE IMPLEMENTATION

Ten equations and twenty-two literal excerpts trace each number to—or away from—its consumer

The equations separate stored defaults, browser derivation, local motion, and program-owned settlement. Every source excerpt is a continuous substring of the reviewed repository. The live observation retains raw account bytes from one shared confirmed Devnet context, while source behavior and deployed program evidence remain separately scoped.

The consumer test

Effect(v, a) = changed outcome only if a production consumer for action a reads value v; storage(v) or display(v) alone does not imply Effect

This is a reasoning rule, not executable syntax. It asks us to name the browser, Guardian, or contract function and the concrete outcome before calling a stat active.

v
The stored or derived value being investigated.
a
The action whose movement, output, permission, or settlement may change.
consumer
Reachable production code that uses v in the outcome for a.

Smelting and Forging are program calculations

B=max(1,batchMultiplier); E=RecipeInputPerBatch×B; R=RecipeOutputPerBatch×B; SmeltOut=max(1,floor(floor(floor(R×ActualInput/E)×RecipeYieldBps/10,000)×SmeltingSkillBps/10,000)); ForgeDurability=min(Candidate(materials,grade,itemLevel,forgingLevel),MaterialCap(materials,forgingLevel))

B expands both expected batch input and recipe output before the actual-input ratio is applied. The nested floors reflect integer division; the compact expression still omits validation, saturation, and the u32 ceiling. Neither accepted result trusts the browser's view-only Smelting bps or simple Forging bonus field.
