PLAYER, WALLET, AND CONTROL · STEP 6

PlayerProfile PDAs: Identity, Equipment Pointers, and Derived Skills

A NiceChunk player is not stored in one all-purpose account. The current source tree keeps a compact PlayerProfile, a much larger Appearance record, a separate unique-name claim, Backpack records, program-scoped progress, and browser-only state. This guide explains a forge-and-market identity limitation found in that source. Its separate dated snapshot covers confirmed Player Program accounts only; it does not prove that deployed Backpack or Market code behaves identically.

Key points

PlayerProfile is an anchor, not the whole character

Its 773 bytes do not contain Backpack inventory, full avatar geometry, a unique-name account, both progress accounts, wallet balance, hotbar selection, or most Profile-screen skills.

The current forged-item pointer is not item-unique

Current Backpack forge source writes the owner's PlayerProfile address into every new forged record's item_pda field. Different outputs can therefore share one key, and the equipment-source Market lookup stops at the first item-kind Backpack record whose key matches. The retained Player-only Devnet snapshot does not prove the deployed Backpack or Market binary follows this source.

The Profile screen is a composite view

It combines Profile and Appearance reads with wallet data, current runtime position, a cache-first Backpack lookup, selected hotbar state, local counters, browser storage, and client calculations.

Program scope decides which progress was actually read

Chunk and Game derive separate PlayerProgress addresses. The current refresh reads only the Chunk-scoped account, so Game-scoped Smelting XP can be missing even when settlement succeeded.

PROGRAM-SCOPED IDENTITY

One wallet resolves to several records with different jobs

The active website configuration points at Devnet. Profile, Appearance, UsernameIndex, and Session belong to the Player Program. Precision Gathering and Exploration progress belong to the standalone Chunk Program. Backpack, Smelting, and Market are currently routed through the unified Game Program. A PDA includes its program domain, so identical seed words under Chunk and Game still produce unrelated addresses.

PlayerProfile is the compact owner-and-world anchor. PlayerAppearance carries visible identity and model data. UsernameIndex carries the canonical name claim. Backpack carries dense inventory records. PlayerProgress carries skill history under the program that updated it. These records can repeat the same owner or Profile key, but no program automatically joins them into one database row.

For an ordinary player, the useful rule is simple: one wallet can point to several public boxes, and each box answers a different question. Reading one box does not prove the other boxes exist, are current, or agree.

Implementation boundary The owner key is a relationship, not a universal account address

COMPACT ACCOUNT, COMPOSITE SCREEN

PlayerProfile stores public fields and checkpoints, not the whole Profile screen

Current initialization writes the NCKPLY01 magic, version 7, owner, GlobalConfig, world ID, position zero, and six defaults: health 100, energy 100, stamina 100, mining power 1, build power 1, and defense 0. It also creates nine empty equipment pointer fields, an empty equipped-Backpack pointer, timestamps, zero forging history, and a bounded name region. No current Player instruction changes those six default attributes.

Position is three signed 32-bit integers. The Player Program checks only Y against Core's minimum and maximum build bounds. Current Play does not read Profile position when choosing a spawn and does not save ordinary walking to chain. Query pose, local position storage, or the default Guardian start chooses Play startup instead.

The confirmed resource-mining path is the current place that may save a Profile checkpoint. If the verified mine transaction did not already carry that save, Play can request a forced supplemental position update. This makes Profile position a sparse chain checkpoint, not a birth coordinate, live tracker, or proof of where the avatar is standing now.

The visible Profile panel then adds information from elsewhere: its level averages effective client skill levels, reputation uses local confirmed-action counters, region comes from runtime position, balance comes from the wallet, and stat cards come from client skill effects. These cards are useful, but they are not direct decodes of the six fixed Profile attributes.

Initialized attributes 100 · 100 · 100 · 1 · 1 · 0

NAME CLAIM, VISIBLE IDENTITY, AND MODEL CODE

A unique name claim and a visible appearance follow separate rules

PlayerProfile reserves 300 bytes for a name, but the contract accepts at most 32 characters drawn from underscore, ASCII letters or digits, and U+4E00 through U+9FFF. The browser precheck uses the broader Unicode Han script class, so some extension characters can pass the browser and still fail the program. For a nonempty name, the program hashes the UTF-8 bytes after lowering ASCII A through Z only and derives UsernameIndex from that hash.

Changing a Profile name does not close the old UsernameIndex, and clearing the name does not release earlier claims. A case-only rename reaches the same index and validates ownership, but an existing index is not rewritten with new display capitalization. A string visible in Profile or Appearance is therefore not, by itself, proof that the matching canonical index exists.

Appearance upsert is the complete visible-identity path: it writes the validated name into Profile, claims or reuses the current UsernameIndex, and packs display name, title, model kind, and model code into Appearance in one instruction. A name-only update changes Profile but not Appearance. Current Play prefers Appearance.displayName, so an older Appearance value can continue masking a newer Profile-only name.

The modelCode contract is deliberately weak: it accepts 3 through 2,048 valid UTF-8 bytes beginning with the case-sensitive prefix NCM. It does not validate an NCM version, decode geometry, bind modelKind to a body, or require a canonical villager. The current Chunk.js loader accepts built-in aliases and renders both NCM2 and NCM4. A prefix-only value such as NCM can pass the program and still fail rendering.

The player funds Appearance storage, but only the configured treasury can close that account and receive its lamports. PlayerProfile and UsernameIndex have no close instruction. Storage funding should not be presented as an owner-controlled refundable deposit.

  1. Validate the exact contract character set

    Apply 32-character, 300-byte, ASCII, underscore, and U+4E00–U+9FFF limits before submission; the browser's broader Han test is not final authority.

  2. Claim or reuse the canonical name address

    A nonempty name needs the SHA-256-derived UsernameIndex to be absent or already bound to the same owner and current Profile.

  3. Update Profile and Appearance together when visible identity changes

    The full Appearance upsert keeps the two records aligned; a name-only write intentionally leaves Appearance untouched.

Implementation boundary Stored model text is not rendered geometry

The chain proves only the weak UTF-8 and NCM-prefix envelope. Chunk.js must still decode an accepted alias, NCM2 body, or NCM4 body before a mesh exists.

POINTER, RECORD, AND CUSTODY

Current equipment pointers can collapse several forged items into one key

set_equipment_slot is not a simple two-field write. The authority must sign; Profile must be writable, owned by Player, be the authority's current player-v7 PDA, and bind the supplied Core-owned GlobalConfig. The Profile slot must be valid. For a real item, Backpack must be owned by either the accepted standalone Backpack Program or unified Game Program, have the current layout and player owner, and equal PDA(account owner, backpack, player wallet, LE64 backpack ID). The selected dense record must also pass forged-item quantity, ID, durability, grade, level, and quality checks.

After validation, Player copies only the record's 32-byte item_pda field into one Profile slot. Backpack remains unchanged: the item is not transferred, reserved, removed, or locked. The same 32-byte value can be written into several Profile slots. Player accepts two possible Backpack account-owner programs, while one compiled Market build accepts only its configured Backpack program owner.

The field name suggests an item-specific address, but the current audited forge source passes player_profile.key into every new output. Backpack then stores that value as item_pda. Different forged records made for the same owner can therefore share the same PlayerProfile public key even when item ID, design hash, grade, level, durability, and Backpack index differ.

Market's two listing sources fail differently when that key is shared. For an equipment-slot listing, Market reads the selected Profile slot, scans Backpack from dense index zero, and immediately validates the first item-kind record whose key matches. If valid, that Backpack record is removed and the Profile slot the player selected is cleared; if that first matching record is invalid, the instruction errors instead of continuing to a later equal-key record.

For a direct Backpack listing, Market removes the Backpack index the player selected, then scans Profile slots from slot zero and clears the first pointer with the same key. Shared keys can therefore make the equipment path remove a different Backpack record than the player expected, or make the Backpack path clear a different matching Profile pointer. These are current audited source-code boundaries; the retained Player-only Devnet snapshot does not prove a matching deployed Market binary or live listing result.

PlayerAppearance reserves twelve 576-byte rich equipment records, but every current Appearance pack writes them empty. The chain decoder reads up to nine Profile equipment pointers, yet current Profile UI renders selected hotbar state and a synchronized Backpack instead. The generated workshop image explains reference versus custody only; it does not prove slot counts or item uniqueness.

Equipment-source Market lookup can therefore fold distinct forged records together. This Player-only Devnet snapshot does not verify the deployed Backpack or Market binaries.

TEN VISIBLE SKILLS, FOUR CHAIN-BACKED CONCEPTS

Progress is split across programs and then merged with browser history

The Profile panel defines ten skills. Precision Gathering and Exploration advance in Chunk PlayerProgress. Smelting advances in a separate PlayerProgress under the unified Game Program's Smelting namespace. Forging XP, forged count, best grade, and best item level live in PlayerProfile. Burden, Craftsmanship, Swiftness, Stamina, Strength, and Appraisal have no matching chain XP field in the current programs.

The browser derives XP from local activity counters and loads one browser object. It first tries a wallet-or-profile-scoped localStorage key; only when that value is absent does it use the legacy unscoped key. Those two browser objects are not merged with each other. The selected browser object is then max-merged with derived counters and the fetched chain subset, skill by skill.

Current refresh calls fetchPlayerProgress once under gameContext.chunkProgramId and requires the account owner to be the Chunk Program. It decodes the Smelting offset from that Chunk account even though successful Smelting settlement writes the identically laid-out PDA under gameContext.smeltingProgramId, currently the unified Game Program. Real Smelting XP can therefore exist while the Profile panel's fetched source remains zero or stale.

Precision, Smelting, and Exploration client threshold formulas reproduce their chain arrays, but Forging does not. Backpack treats 45,000 forging XP as level ten; the generic UI curve shows level four at that value. Backpack also calculates gained forge XP as 25 times final grade squared. Player does not recompute that equation: it validates the forging authority and directly saturating-adds the nonzero gained_xp supplied by Backpack.

A large displayed value can therefore be honest about the browser's merge rule without being chain truth. Client-only effects may use that visible maximum, while mining and Smelting settlement still enforce their own program rules.

SOURCE-BY-SOURCE PROVENANCE

Treat every Profile card as a claim with its own source

The current chain decoder reads Profile position, fixed attributes, nine equipment pointers, equipped Backpack, forging history, and name. The UI does not consume every decoded field. Its equipment panel uses the selected hotbar record for the main hand and synchronized Backpack data for the back slot, leaving the other conceptual slots empty or locked.

Backpack synchronization is cache-first. If a wallet-scoped cached Backpack address still loads as a valid owner Backpack, the helper returns it before reading Profile.equippedBackpack. Only a missing or invalid cache falls back to the current Profile pointer. A screen can therefore show a valid Backpack without proving that it was freshly reconciled with the current Profile selection.

Identity is also layered. Appearance display name wins over Profile name; local profile values and built-in villager aliases can supply fallbacks. Runtime position, wallet balance, local counters, and derived skill effects supply other cards. The current Profile panel has no visible control wired to the available savePlayerName or savePlayerAppearance helpers, so helper existence is not proof that players can edit identity there.

The browser chain module and exported TypeScript SDK do not currently agree on every name instruction. Static source shows the SDK building three accounts for a nonempty name update where the processor requires five, and five accounts for Appearance upsert where the processor requires six. The browser module constructs the five- and six-account forms. Existing tests assert older SDK lists, so passing those tests does not establish deployed-contract parity. No unsupported Devnet simulation result is claimed here.

A trustworthy Profile screen does not need one giant account. It needs labels and reconciliation: name the account or local source, fetch progress under the correct program, distinguish Backpack custody from Profile pointers, and keep local-only values visibly local.

  • Identity: state whether Appearance, Profile, local profile, or built-in fallback won.
  • Backpack: state whether the value came from wallet cache or a fresh Profile pointer fallback.
  • Equipment: separate Backpack record, Profile pointer, empty Appearance record, and selected hotbar presentation.
  • Skills: show Chunk, Game, Profile, derived, and browser sources instead of silently calling their maximum chain XP.
  • Position: distinguish runtime location, local startup save, and sparse chain checkpoint.
  • SDK: compare constructed account lists with the processor contract before treating a test as integration proof.
What the code guarantees Provenance is more useful than an on-chain badge

A precise label such as Appearance, Profile, Game progress, wallet cache, runtime, or local derived value tells a player what was actually observed.

RETAINED CONFIRMED DEVNET SNAPSHOT

Current and earlier Profile generations still coexist

At confirmed context slot 476,758,441 on 16 July 2026, the retained audit captured all 47 Player-Program-owned accounts with raw Base64 data and hashes. Nineteen carried Profile magic: eleven matched the current owner-derived player-v7 PDA, while eight were earlier records. The remaining recognized accounts were ten Appearances, seven UsernameIndexes, ten Sessions, and one InviteIndex page.

The eight earlier Profile records consisted of four 473-byte version-2 records, three 773-byte version-3 records, and one 773-byte version-4 record. Length alone is therefore insufficient: four earlier records use the current 773-byte length. Current writes derive player-v7, so the canonical owner-derived address, version, layout, owner, and GlobalConfig all matter.

Among the eleven current Profiles, ten had moved away from origin, five stored an equipped-Backpack pointer, none stored any nonzero value in the nine equipment fields, and two had forging history. All 120 reserved rich equipment states across ten Appearances were zero. All ten retained Sessions were expired at the snapshot and had action count zero. These observations show which layouts are live; they do not prove unused paths are impossible.

The full upgradeable-loader ProgramData account was 172,925 bytes: 45 bytes of loader metadata followed by a 172,880-byte ELF allocation. That ELF allocation hashed to 9158af26bb95eea3848b10ab06dbee822f8c0f30552e7538b494ac6a2ef9f32a and matched the retained local 172,880-byte artifact. The byte match does not prove that the dirty source worktree reproducibly built it.

The archive artwork deliberately does not reproduce these counts. Boxes can make old and current records intuitive, but only the retained audit establishes the dated numbers, raw account bytes, hashes, context slot, and interpretation boundaries.

function loadJsonObject(primaryKey, fallbackKey) {
  try {
    const storage = globalThis.localStorage;
    if (!storage) return {};
    const raw = storage.getItem(primaryKey) || storage.getItem(fallbackKey);
    const parsed = raw ? JSON.parse(raw) : {};
    return parsed && typeof parsed === "object" && !Array.isArray(parsed) ? parsed : {};
  } catch {
    return {};
  }
}

function mergeSkillXp(...sources) {
  const result = {};
  for (const source of sources) {
    if (!source || typeof source !== "object") continue;
    for (const [key, value] of Object.entries(source)) {
      const numeric = Math.max(0, Math.round(Number(value) || 0));
      result[key] = Math.max(result[key] || 0, numeric);
    }
  }
  return result;
}
