Field Guide / Material Lifecycle

FOUNDATION 06 · RESOURCE OWNERSHIP

A material counts only after every handoff accepts it

NiceChunk cotton begins as a deterministic PDA rule bound to an eligible grass face, not as a free-standing inventory voxel. Mining stores the supporting terrain block first and adds cotton only when backpack capacity remains. Recipe 1025 then matches exactly four cotton blocks and one dry-grass block, requires a separate fuel slot, consumes the selected records, and creates one volume-bearing cotton-cloth material for the forge.

11 min read

Key points

Deterministic does not mean already owned

The same seed, coordinate, ordered PDA rows, and rule revision resolve the same cotton candidate. Ownership begins only if the successful mining transaction actually appends the cotton reward to the player's backpack.

The visible reward is deliberately lossy

The mined grass record is strict, but cotton and an exploration extra are secondary records. Cotton has priority; either secondary reward is silently skipped when no slot remains.

Recipes and forging consume whole selected slots

The recipe program verifies identifiers, distinct indexes, heat, and output capacity, then replaces the inputs with cloth and calculates bounded output volume inside the same atomic transaction. The forge later verifies aggregate material capacity, not partial per-voxel consumption or dye provenance.

SURFACE DECORATION TABLE

Cotton begins as one ordered rule bound to a grass top face

The current SurfaceDecorationTable revision contains forty-three generic numeric rows. Rule 74 maps canonical grass block 1 to cotton drop block 48, uses decoration model 12, occupies roll interval 425 through 454 for salt 201, and carries the shadow and mineable flags. Chunk.js fetches those rows for rendering; the Chunk Program reads the PDA again when it evaluates a reward.

A coordinate must first resolve an uncovered canonical grass surface inside the rule's height range. Water above the surface suppresses the decoration, and a generated tree voxel in the bound space wins instead. Rows using the same salt share one hash roll and are evaluated in stored order, so cotton and the five flower colors occupy disjoint bands rather than running six unrelated random trials.

A stable 0.3% interval among eligible grass coordinates, not a repeated swing chance.

Implementation boundary The plant is not an independently mined world voxel

Cotton is a face-bound decoration resolved from the supporting coordinate. The canonical mining action targets the generated grass beneath it; the program resolves the current PDA rows for that coordinate and considers cotton only if the resulting entry matches the mined surface.

ONE ATOMIC MINING ACTION

The program rechecks the support, records the break, and stores grass before cotton

A mining transaction supplies integer coordinates and the client's expected base block. The Chunk Program verifies the session and canonical accounts, rejects a foundation-protected coordinate, regenerates the base block, reads both rule PDAs, and rejects air, water, bedrock, or an expected-block mismatch. A decoration becomes eligible for the secondary reward only when its surface Y and surface block match the block being mined; a mismatch removes that extra without invalidating the base mining action.

The transaction records the broken local coordinate and then invokes the Backpack Program for the required terrain reward. Only after that strict append succeeds does it build a secondary list. Cotton is first, with block ID 48, the supporting coordinate, the same precision-gathering volume, and metadata packing rule ID 74 with decoration ID 12. An optional ResourceDropTable exploration reward follows it.

  1. Reconstruct

    Resolve the canonical block and PDA-driven decoration for the submitted coordinate.

  2. Stage the mandatory state

    Insert the broken coordinate and strictly append the mined support block to the owner's backpack.

  3. Offer ordered extras

    Try cotton first and an exploration drop second without making either one a reason to fail the base mining action.

What the code guarantees A full failure rolls back the broken coordinate too

The base backpack append is part of the same Solana transaction as the ChunkBroken update. If the mandatory grass record cannot be stored, the transaction fails atomically; it does not leave a mined hole with no base reward.

BACKPACK CAPACITY

Cotton is visible in the world but still needs a second free slot

The backpack account physically reserves ninety-nine 80-byte records, but current initialization limits a new backpack's active capacity to at most fifty. Item count is dense: records occupy indexes zero through itemCount minus one. The strict grass append increases that count before the lossy batch sees the account.

If mining begins with exactly one free slot, grass occupies it and the lossy batch returns success without cotton. If two slots were free, cotton fills the second and an exploration extra is skipped. There is no pending mailbox, later claim, or compensating mint. Players therefore need at least two open slots before breaking a cotton-bearing grass face if they want both the support block and cotton.

Current initial ceiling 50 active slots

The account format reserves 99 records, but the initialization instruction clamps capacity to 50.

One free slot before mining Grass only

The mandatory support reward fills the backpack; cotton is silently skipped.

One free slot after grass Cotton wins

The visible decoration precedes the invisible exploration extra in the lossy batch.

How to read this Successful mining does not prove every eligible reward was retained

The accepted transaction proves the base break and whatever backpack records actually fit. It does not promise that a capacity-skipped secondary reward can be recovered later.

RECIPE 1025

Four cotton records and one dry-grass record must match exactly

The current enabled recipe in table 222 has five block inputs in order-independent matching: cotton IDs 48, 48, 48, 48 and dry-grass ID 29. Its chain output is material item code 1025. The public material catalog maps that code to class fiber, forge use fabric, and render mode cloth. Plant fiber is no longer in the public material catalog; historical recipes 1003 and 2003 remain disabled so old numeric identities do not shift.

Execution must also include at least one fuel index whose maximum heat tier reaches the recipe's minimum of one. Input and fuel indexes share one uniqueness set, so the dry-grass record counted among the five recipe inputs cannot double as fuel. A batch multiplier repeats all five recipe rows, but it still produces one material slot whose volume is calculated from that output template and the batch's actual-to-template input-volume ratio.

Verified input IDs 48 + 48 + 48 + 48 + 29

The chain matches packed block identifiers from authoritative backpack records.

Output identity Item code 1025

One cotton-cloth material record is appended for the current single-output recipe.

Heat boundary Tier 1 · separate fuel

At least one unique fuel slot is required in addition to the five recipe inputs.

Implementation boundary The source data's one-million cubic millimetres is not a fixed payout

The output template supplies a one-million-cubic-millimetre baseline, not a guaranteed payout. Live execution scales that baseline by the selected-versus-expected input volume ratio and batch multiplier, then applies the 6,500-basis-point fiber yield and the player's smelting level.

AUTHORITATIVE MATERIAL TRANSITION

Recipe execution removes the selected slots and writes the resulting cloth volume

Before mutating the backpack, the Smelting Program validates its owner, capacity, every unique input and fuel index, exact multiplicities, heat tier, and the room available after removal and output insertion. It sums the recipe-input volumes, excluding fuel, then removes every selected input and fuel record through the Backpack Program.

For each active output, the program starts from that output's PDA template volume, scales it by batch multiplier and by the ratio of actual selected input volume to expected recipe-template input volume, then applies recipe yield and the current Smelting skill factor with integer arithmetic. For one default-volume cotton-cloth batch, the one-million template becomes 65,000 cubic millimetres at level zero or 390,000 at level ten. The current one-output recipe writes one cloth slot carrying that calculated batch volume.

Fiber recipe yield 6,500 bps

Applied to the scaled per-output PDA template before the skill factor, using integer division.

Skill output factor 1,000 to 6,000 bps

Level zero starts at 10%; levels one through ten add 5 percentage points each.

Mutation scope Inputs + fuel out · cloth in

All records change inside one atomic transaction.

What the code guarantees This transition consumes real backpack state

Unlike a local crafting preview, an accepted recipe transaction removes the exact authoritative indexes it validated and appends the output record. A later failure rolls back both sides of that replacement.

CLOTH INSIDE THE FORGE

The editor inherits rich material behaviour; the chain keeps a narrower aggregate proof

The forging page maps item code 1025 back to cotton cloth and asks Chunk.js for the baked cloth surface used by material swatches and the live workbench. Locally, each 14 by 10 by 14 component measures occupied cells, scales the selected slot's volume to the material actually present, multiplies that volume by density score, and uses the resulting mass weight across twelve attributes. Restored equipment rebuilds encoded geometry and RGB444 colors, then uses its cloth identity for wind motion; it does not retain the workbench material tile. Cloth painting and runtime flutter are presentation and design behaviour, not a chain simulation of fabric.

Forge submission takes whole selected material slots. The Backpack Program verifies that every index is a valid material, rejects duplicates, checks that total selected volume and effective durability meet the NCF1 requirements, consumes all selected slots, and appends one forged record. It does not persist the full input list, twelve-attribute spectrum, exact cloth provenance, or dye-to-paint relationship; the forged record keeps aggregate outcome fields and a 32-bit design hash.

Five flower-dye recipes are enabled, but the current verified world path has no native way to collect water block 17: water is unmineable, rule tables reject water drops, and no bucket instruction exists. The page therefore treats dye as an available recipe definition, not a complete world-harvest-to-dye loop.

Chain forge checks Volume + effective durability

The current verifier checks two aggregate capacities across whole selected slots.

Dye status Recipes exist · collection loop incomplete

Implementation boundary A painted cloth design does not prove a matching dye was consumed

The official editor can map cloth and palette choices into NCF1, but the current forge instruction validates material slots and aggregate capacities. It does not bind a paint color to a particular dye item code.

DROP, CAPACITY, AND YIELD MATH

One fixed hash chooses the plant; ordered slots decide whether the reward survives

The first equation scopes cotton to eligible grass coordinates. The second models the strict base reward and ordered lossy extras. The third mirrors recipe 1025's exact input counts, per-output PDA volume baseline, selected-input ratio, and two-stage integer yield.

Cotton's deterministic rule interval

H = surfaceHeight(x,z); r = hashCoord3(seed,x,H+1,z,1200+201) mod 10000; cotton iff surfaceBlock=grass AND (waterLevel is null OR waterLevel<=H) AND treeBlock(x,H+1,z)=air AND 425<=r<455; P_roll(cotton | eligible grass) = (455-425)/10000 = 0.003

Rule 74 occupies thirty values in the shared ten-thousand-value band for salt 201. The 0.3% figure is the conditional hash share for an otherwise eligible grass face; it is not a fresh random chance per swing and not a percentage of every world block.

H
Canonical generated surface height at horizontal coordinate x,z.
r
Stable unsigned rule roll shared by rows using salt 201 at this coordinate.
425 <= r < 455
The half-open interval assigned to cotton in the current revision-2 table.

Strict support reward, then ordered lossy extras

F = max(0,C-n-1); A_decoration = D*1[F>=1]; A_extra = E*1[F>=1+D]; n_after = n+1+A_decoration+A_extra

For a mining transaction with room for its mandatory base block, n is the occupied count before mining and C is capacity. The +1 stores the supporting terrain block. D and E indicate an eligible surface decoration and exploration extra. When only one slot was free before mining, grass fills it and cotton is skipped; when one slot remains afterward, cotton takes it before the exploration reward.

F
Free slots remaining after the required mined terrain record is appended.
D, E
Binary indicators for an eligible surface decoration and exploration extra.
A_decoration, A_extra
The secondary records that actually fit and are therefore appended.

Cotton-cloth inputs and per-output integer volume

N_cotton=4m; N_dryGrass=m; N_fuel>=1; inputIndexes intersect fuelIndexes = empty; b_skill=min(6000,1000+500L); V_expected=max(1,m*V_recipe); V_scaled=floor(max(1,V_template)*m*max(1,V_in)/V_expected); V_cloth=max(1,min(2^32-1,floor(floor(V_scaled*6500/10000)*b_skill/10000)))

Batch multiplier m repeats each of recipe 1025's five input rows, while every input and fuel index must remain unique. Fuel volume is not part of V_in. The program first scales this output's one-million-cubic-millimetre PDA template by the ratio between selected and recipe-template input volume, then applies the 6,500-basis-point recipe yield and the 1,000-to-6,000 skill factor with integer division. It does not divide one aggregate result evenly among outputs.

m
Recipe batch multiplier; one batch selects four cotton blocks and one dry-grass block.
L
Current smelting level from zero through ten.
V_recipe, V_template, V_in, V_cloth
One batch's expected recipe-input volume, this output's PDA template volume, selected recipe-input volume, and the resulting cloth-record volume in cubic millimetres.
