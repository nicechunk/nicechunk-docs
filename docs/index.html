Status Model

Every rule must say whether it is live, planned, client-only, or intended for chain verification.

This is the core documentation rule for NiceChunk. A feature can be playable before it is fully decentralized, but the page must say which part is already live and which part is still a protocol target.

Live Alpha Playable client behavior

Browser world, seeded terrain, movement, mining interaction, placement preview, hotbar, backpack UI, forging UI, minimap, and wallet login.

Devnet / Chain Explicit Solana state

GlobalConfig, player/session bridges, Guardian registry and region accounts, NCK mint binding, and PDA derivation rules.

Planned Protocol Future verification layers

Chunk override compression, resource settlement, asset ownership, marketplace settlement, Guardian proofs, and governance controls.

Trust Boundary No hidden claims

The docs must not imply that client-only rendering, local cache, or unreleased systems are already fully decentralized.

Implementation Snapshot

Current docs should describe the game that actually exists today.

This snapshot separates live browser behavior, devnet-backed state, and protocol targets so players can understand what is authoritative now and what still belongs to the roadmap.

Live Alpha Infinite seeded world

Terrain is generated on demand from deterministic rules. The client loads nearby chunks, keeps discovered map information, and unloads render resources outside the active area.

Live Alpha Water as blocks

Rivers, lakes, oceans, and dynamic water visuals use water block entries. Water is shorter than a normal block, ignored by collision, blocks mining selection above covered cells, and uses depth-based color.

Devnet / Chain Player and backpack PDAs

Realtime Guardian access layer

The browser can connect to Guardian realtime endpoints for nearby player movement, chat, and dig events. Guardians provide access and visibility; they do not custody assets or sign player transactions.

World Generation

The same seed and rule version must rebuild the same world.

Terrain is generated from deterministic inputs. Large climate regions are sampled first, then oceans, dunes, foothills, mountains, snow bands, trees, and resource hints are layered on top.

Deterministic Seed contract

A public seed is normalized into numeric streams. Terrain, climate, tree palettes, and resource hints must use seed-derived streams.

worldSeed = hash(seed)
height = terrain(worldSeed, chunkX, chunkZ, x, z)
Implemented Slope-limited mountains

Mountain growth is broad and probabilistic. Final terrain limits neighboring columns to natural one-block steps where gameplay requires traversal.

nextHeight = rawHeight(x, z)
height = min(nextHeight, neighborHeight + 1)
Inspectable Large biome regions

Low-frequency climate masks allow large plains, deserts, ocean bodies, forests, foothills, and snow regions without hard size caps.

temperature = fbm(x * 0.0032, z * 0.0032)
moisture = fbm(x * 0.0038, z * 0.0038)

Water System

Water is a block state first, then a visual surface.

The terrain generator and dynamic water layer treat water as cell data. Rendering can make water translucent and shorter, but gameplay checks must derive from block state instead of screen pixels.

Generated Generated water blocks

Sea, lake, and river cells resolve to water-family block ids. Their top should never render above the surrounding ground plane.

Collision Boundary No solid collision

Player collision ignores water cells so the character naturally falls to the real solid block underneath instead of standing on water.

Selection Rule Water blocks mining

If water covers a target column, mining selection is blocked. This prevents players from standing in water and mining hidden blocks underneath.

function resolveWaterCell(x, y, z) {
  const generated = generatedBlockAt(x, y, z);
  const dynamic = dynamicWaterAt(x, y, z);
  const placed = placedBlockAt(x, y, z);

  if (placed) return placed;
  return dynamic ?? generated;
}

function collisionBlockAt(x, y, z) {
  const block = resolveWaterCell(x, y, z);
  return isWater(block) ? null : block;
}

function canMine(target) {
  return !waterCovers(target.x, target.y, target.z)
    && isBreakable(target);
}

Chunk System

World coordinates become chunk coordinates, local block coordinates, and cache keys.

A chunk is a 16 x 16 X/Z area. The client currently renders a 13 x 13 visible chunk square and preloads a wider 19 x 19 square to reduce stalls. Rendering resources can unload while discovered minimap data stays cached.

Coordinate Demo World To Chunk 
const CHUNK_SIZE = 16;

function worldToChunk(worldX, worldZ) {
  return {
    chunkX: Math.floor(worldX / CHUNK_SIZE),
    chunkZ: Math.floor(worldZ / CHUNK_SIZE),
    localX: mod(worldX, CHUNK_SIZE),
    localZ: mod(worldZ, CHUNK_SIZE)
  };
}

function chunkKey(worldX, worldZ) {
  const chunkX = Math.floor(worldX / CHUNK_SIZE);
  const chunkZ = Math.floor(worldZ / CHUNK_SIZE);
  return `${chunkX},${chunkZ}`;
}

function chunkCommitment(seed, chunkX, chunkZ, ruleVersion, overrideRoot) {
  return sha256(`${seed}:${chunkX}:${chunkZ}:${ruleVersion}:${overrideRoot}`);
}

On-chain State

The chain stores commitments and player-owned state, not every generated block.

NiceChunk’s storage model is designed around one rule: deterministic generated terrain should be free to reconstruct, while changed or economically meaningful state becomes explicit and verifiable.

Player & Backpack Contracts

Buying a backpack and equipping a backpack are separate state changes.

The Backpack program creates the inventory account. The Player program binds that account to the player's public profile. The profile panel only displays a backpack after the equipped backpack field in PlayerProfile points to a valid Backpack account owned by the same wallet.

Purchase Flow Required instruction sequence
  1. 01

    Create PlayerProfile if it does not exist.

  2. 02

    Create Backpack PDA with seed ["backpack", owner, backpack_id].

  3. 03

    Call Player program instruction 5 to write the Backpack public key into PlayerProfile.

  4. 04

    Reload PlayerProfile, then load the referenced Backpack and verify both accounts share the same owner.

function loadEquippedBackpack(owner) {
  const profile = readPda(["player", owner]);
  if (profile.equippedBackpack === DEFAULT_PUBKEY) return null;

  const backpack = readAccount(profile.equippedBackpack);
  if (backpack.owner !== owner) return null;

  return backpack;
}

Performance & Storage

The infinite world stays cheap by deriving first and storing only changes.

NiceChunk should not pay per generated block and should not add per-frame logic for rules that can be resolved at generation, chunk-build, click, or transaction time.

Render Time Batch by chunk

Blocks, water, trees, and decorations are built into chunk meshes or instanced batches. Far chunks use cheaper detail modes instead of per-object realtime decisions.

Algorithmic No visual detection for logic

Collision, water coverage, mining selection, and path checks should use generated block state and local deltas, not browser screenshots or pixel inspection.

PDA Storage Sparse changed cells

The default world is reconstructed from seed. PDA data should hold compact deltas, mined masks, or section patches only where players changed the world.

Ephemeral Dynamic water stays bounded

MVP flowing water is visual or client-side. Future authoritative water should store compact source changes, not full flowing grids.

// A 16 x 16 binary layer can be represented as a 256-bit mask.

Player Actions

The selected hotbar slot is the action contract.

NiceChunk actions should be predictable before they are submitted. The client resolves input based on the held item, then range and target rules decide whether an action is valid.

Empty hand

Clicking with no held item produces a swing only. It does not mine or place.

Pickaxe

A mining tool resolves clicks into mining attempts and consumes durability when successful.

Block item

A block stack resolves clicks into placement attempts within range and consumes one item per successful placement.

type HeldItem =
  | { type: "empty" }
  | { type: "tool"; toolId: "iron_pickaxe"; durability: number }
  | { type: "block"; blockId: string; amount: number };

function resolvePrimaryAction(item: HeldItem) {
  if (item.type === "tool" && item.toolId === "iron_pickaxe") return "mine";
  if (item.type === "block" && item.amount > 0) return "place";
  return "swing";
}

Mining & Resources

Resource outcomes should be reproducible from public inputs.

Basic terrain can be deterministic, but valuable resources must not be fully predictable from public world seed alone. Discovery should combine public commitments with action-specific inputs so players can verify outcomes without pre-scanning every valuable coordinate.

Proof Demo Mining Roll 
function resourceRoll(input) {
  const payload = [
    input.worldSeedHash,
    input.resourceRuleHash,
    input.chunkX,
    input.chunkZ,
    input.blockX,
    input.blockY,
    input.blockZ,

    input.recentSlotEntropy

}

Assets & Market

Tradable items need compact asset data and explicit listing state.

NiceChunk has separate workstreams for compact on-chain 3D assets, CPU-searchable asset genes, and marketplace listings. These systems should remain clearly separated from raw gameplay inventory until ownership and settlement rules are complete.

Tooling NCM asset format

NCM converts voxel models into compact blockchain-readable text so clients can reconstruct 3D assets from account data without an external gateway.

Experiment NCM DNA search

NCM DNA explores CPU-friendly searches for shorter deterministic genes that reproduce the same asset hash and reduce future PDA storage size.

Contract Market listings

The market program creates seller-owned listing PDAs with category, currency, source kind, source index, quantity, price, and item hash. Current scope is listing and canceling.

Guardian Network

Guardians are access infrastructure, not asset custodians.

The current Guardian program registers region endpoints and NCK stake on devnet. The broader Guardian network concept is the realtime access layer that keeps regions reachable without owning player assets or signing player transactions.

Guardian Region Live access concept
BoundaryNo custody

Guardians do not hold player assets and do not replace wallet authorization.

BoundaryNo transaction signing

Players interact with Solana wallets directly. Guardians should not process transactions as hidden owners.

DevnetGuardian registry

GuardianRegion accounts store region coordinates, endpoint, owner, operator, stake, proof counters, and update slot.

MagicBlock ER

High-frequency chunk changes should move through a bounded realtime path.

The MagicBlock Ephemeral Rollups bridge is scoped to one active chunk first. It is a devnet-only direction for fast break/place updates, not a replacement for wallet ownership or final settlement.

Scope Single active chunk

The client should subscribe to the player's current chunk, then unsubscribe and resubscribe when the player crosses chunk boundaries.

Session PlayerSession authority

A wallet signs once to create or refresh a session PDA. The temporary session key can then sign allowed gameplay actions until expiry.

Boundary No broad RPC polling

The design should not re-enable wide multi-chunk polling on public RPC. Realtime updates must stay bounded to active interest areas.

Devnet Not production enabled

Do not enable the browser ER switch for production until the chunk program is redeployed and a chunk delegation is verified end to end.

Transaction Debugging

A signature is not enough; confirmed transactions must be checked for errors.

Wallets can return a signature even when the confirmed transaction later contains an error. Client code and support docs should verify confirmation status, account ownership, and PDA fields before telling a player that state is bound.

Wallet Wallet mismatch

The connected provider must match the stored NiceChunk wallet address. Multiple installed wallets can expose different providers.

Confirmation Check value.err

After confirmTransaction, inspect the returned error field before treating a purchase, listing, mining action, or session refresh as successful.

PDA Read the authoritative PDA

For backpacks, read PlayerProfile first. The profile must point to a Backpack account whose owner equals the wallet address.

const confirmation = await connection.confirmTransaction({

Fairness, Security & Governance

Rules are only fair when players can inspect the rulebook.

NiceChunk should keep deterministic rules public, version major rule changes, publish test vectors, disclose trust limits, and distinguish live features from planned protocol mechanics.

01Deterministic generation

The same seed, chunk coordinate, and generation version should produce the same result.

02Versioned rule changes

UI changes, world-rule changes, economy changes, and chain-state changes should be tracked separately.

03Trust limits

The docs must state what is client-only, what is Guardian-assisted, and what is intended for on-chain verification.
