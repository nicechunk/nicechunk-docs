MARKETPLACE 14 · ESCROWED SLOTS AND ATOMIC DELIVERY

Marketplace Listing PDAs: Escrowed Slots, Split Payments, and Atomic Delivery

NiceChunk Marketplace creation removes one complete 80-byte record from a seller-owned Backpack and stores those exact bytes in a seller-derived Listing PDA. A purchase reads the committed price and currency, splits SOL or NCK between the seller and fixed treasury, appends the unchanged record to a buyer-owned Backpack through the market-authority PDA, and marks the Listing sold in one atomic instruction. The active Devnet website routes Market namespace 4 through the unified Game Program. Its browser still mixes real PDA listings with local-only orders and sample previews, and refreshed chain cancellation is currently broken because Play does not supply the required destination Backpack.

15 min read

Key points

Escrow is the complete Backpack record

Listing version 5 reserves 80 of its 216 bytes for the unchanged source slot. Creation copies that record, removes the whole dense Backpack entry, and clears one matching equipment position when the same item is detected there. There is no amount argument or partial-stack decrement.

Payment and delivery succeed or roll back together

The program reads price and currency from the active Listing, applies a hardcoded one-percent integer fee, transfers SOL or NCK, appends the exact slot through market-authority, and marks the account sold. A full Backpack, invalid token account, arithmetic overflow, or later failure rolls back the Market's earlier payment transfers and program-account mutations; a processed failed transaction may still charge its network fee.

A market card is not necessarily a chain order

Play combines decoded Listing PDAs with browser-local pseudo-listings and sample previews. Local hashes do not prove escrow, samples cannot be bought, user-selected building and clothing categories are not committed, and the current refreshed-chain cancellation call omits the Backpack required by the program.

DEVNET NAMESPACE FOUR

The current marketplace lives inside the unified Game Program

public/mainnet.json currently declares Devnet and assigns the same 6CurnvneezBuHwPUnrCiFg1QMWeUF67ufQxYebyr2UP7 address to Backpack, Smelting, and Market. The Game dispatcher consumes namespace byte four before forwarding the remaining inner instruction to the Market processor. Listing and market-authority PDAs are therefore derived under that unified address in the website context, even though the standalone Market crate still declares 1PwPzFtdJ5gQqku5gBo4b6Wvo48Qe8NuXSogUP8TWpR for non-unified builds.

The current Play HTML contains the Market panel but no element with id marketButton, so the normal button binding resolves to null. Its connected entry is the debug command line: press slash, then run /market or /trade. This discoverability gap does not change any program rule, but it matters when describing what a player can reach through the present production interface.

A Listing address combines the literal listing seed, seller public key, and little-endian listing ID. Creation requires that seller to sign, validates the derived address, rejects an already program-owned account, and charges the seller the rent needed for 216 bytes. Cancellation re-derives the same PDA and requires the stored seller to sign. Purchase re-derives it from the stored listing ID and seller account, requires a different signing buyer, and rejects any seller account that does not match the Listing bytes.

MarketAuthority has no player or listing component. It is a single PDA under the active Market Program and signs the CPI that delivers an escrowed record to a buyer Backpack or returns it to a seller Backpack. The Backpack Program independently derives that authority under its configured Market Program ID, so supplying a look-alike account cannot authorize arbitrary inventory creation.

ONE RECORD LEAVES THE SELLER

Listing creation escrows exact slot bytes, not a category label or partial quantity

ListingAccount version 5 has a fixed 216-byte layout. Its central 80-byte field is a complete BackpackSlotRecord copied before mutation. For a block record, those bytes include kind, quantity, packed world position and block ID, volume, and metadata. For an item, they also carry category, item code, item ID, item PDA, volume, current and maximum durability, grade, level, quality, and metadata. ListingAccount adds seller, currency, base-unit price, source type and index, timestamps, and eventual buyer data around that record.

Backpack source mode validates the supplied account as a seller-owned Backpack PDA, reads the selected dense index, requires a block or item kind and nonzero quantity, then removes that complete record through an owner-signed Backpack CPI. If its item PDA also appears in the PlayerProfile equipment array, creation clears that equipment slot. Equipment source mode instead validates one of nine equipment positions, finds the same item PDA in the seller's Backpack, removes its entire record, and clears the chosen profile position. The current Play sell grid nevertheless submits only source backpack, so direct equipment-source creation exists at the program level but is not exposed by that flow.

There is no listed amount, decrement, or replacement record. A record with quantity ten is escrowed as one complete 80-byte record containing quantity ten. Dense removal shifts subsequent Backpack indexes. The Listing preserves the original source index and whether the request named Backpack or equipment, but it does not preserve the source Backpack public key. It also stores no market category field, so the browser's raw, equipment, building, and clothing selections cannot all survive a chain round trip.

PRICE FROM THE LISTING, SLOT TO THE BUYER

SOL and NCK settlement share one fee rule and one atomic delivery boundary

Purchase first validates an active version-5 Listing, its program owner, its seller, and its seller-derived PDA. The buyer must sign and cannot equal the seller. Currency and price are then read from the Listing itself; the buyer cannot substitute a lower amount in the instruction. NCK uses the fixed Devnet mint HSnWF5kjkWVrceW2SaSskScuLveUZE4gpthZ2ZXRPQPo, transfer_checked with six decimals, and token accounts whose program owner, mint, and wallet-owner bytes are validated. SOL uses System Program transfers and the fixed treasury CtPV2vmqNNwUSfMu5nz58ZtMPy6ZvxL4LyNdPHVW7WvF.

Both branches call split_market_payment. Its private 100-basis-point constant floors the fee after checked multiplication, so the seller receives price minus fee and treasury receives the remainder. The current Core GlobalConfig generator and state constant also say 100 basis points, but Market never receives or reads GlobalConfig during purchase. That equality is configuration coincidence, not dynamic fee authority. Prices one through ninety-nine base units pay a zero fee, and a valid listed u64 above 184467440737095516 cannot later settle because multiplication by 100 overflows.

Only after the seller transfer and any nonzero treasury transfer does Market invoke the Backpack Program with the stored slot and market-authority signature. Backpack validates its PDA and owner, decodes the record, and performs a strict append. If itemCount already equals capacity, BackpackFull aborts. If append succeeds, Market records sold state, buyer, the sale's Solana Clock slot, sold time, and updated slot. All CPIs and the final write belong to one Solana instruction, so failure at capacity, token validation, payment, append, or marking sold rolls back every Market payment transfer and program-account mutation; the transaction's network fee may still remain charged.

The website performs useful preliminary checks for a buyer Backpack, free capacity, and NCK associated token accounts, and may create missing seller or treasury associated accounts in the same transaction. Those checks improve feedback but are not settlement authority. The program's validation and confirmed post-transaction accounts remain decisive.

Current Play has no mutation lock around create, buy, or cancel. After success it refreshes the listing query and rerenders, but the Market integration has no refreshBackpack callback and does not call the Backpack mutation refresh used by Smelting. Confirmed Backpack changes can therefore remain visually stale until the roughly fifteen-second frame poll, while repeated clicks remain possible. During creation, the instruction commits only the dense index. If a concurrent Backpack change leaves that index valid, the program escrows the record currently there, which may differ from the stale preview; if the index becomes out of range, creation fails with invalid escrow inventory.

RETURN, ERASE, OR PERSIST

Cancellation closes an active Listing, while the current instruction set leaves sold rent locked

Only the stored seller may cancel an active Listing. After validating seller, account owner, Listing bytes, and PDA, the processor requires a destination Backpack, Backpack Program, and market-authority account. The destination is not recovered from ListingAccount because that address was never stored. It may be any valid seller-owned Backpack PDA with free capacity. The exact escrowed slot is appended there before account closure; a full or invalid Backpack rolls the cancellation back and leaves the Listing active. Cancellation receives no PlayerProfile account and does not restore the item's former equipment position.

Successful cancellation writes the canceled state byte and immediately calls data.fill(0), then transfers every Listing lamport to the seller and sets the Listing balance to zero. The intermediate state is therefore not durable queryable history. On 2026-07-15, the Devnet rent-exemption query for 216 bytes returned 2,394,240 lamports, or 0.00239424 SOL; that mutable amount was the gross Listing refund at audit time, before the seller fee payer's transaction fee.

Purchase has a different lifecycle. mark_sold retains the complete 216-byte account and its lamports. Cancellation rejects non-active state, processing tag three is disabled, and no other close or cleanup instruction exists. A sold seller-and-ID address therefore remains allocated, its rent remains locked, and the same listing ID cannot be initialized again at that PDA.

The current Play chain cancellation path cannot satisfy the five-account route. Refreshed decoded listings expose sourceInventory as null, and cancelChainListing passes only the Listing address and ID. The chain helper adds destination accounts only when sourceInventory is supplied, so it constructs a two-account inner instruction that Market rejects with InvalidAccountCount. Browser-local cancellation is a separate attempted snapshot-restoration path, not a Listing PDA cancellation, and the current PDA-only GameState does not normally permit that restoration to complete.

The account must have room; the original Backpack address is not committed.

Sold history Persistent, no close route

Implementation boundary Chain cancellation is currently broken in the Play interface

Do not tell users that the current refreshed-order Cancel button can return escrow. The program requires a destination Backpack, while Play currently omits it and submits the wrong account count.

PDA ACCOUNTS, LOCAL ORDERS, AND SAMPLES

Browse combines three different states that must not be called equivalent

The chain helper asks the unified Game Program for 216-byte accounts and may add RPC memcmp filters for seller, active/sold state, and currency. It accepts decoded accounts after length and Listing-magic checks, without rejecting another version or invalid enum at display time, then applies category, source, text query, sorting, and pagination in JavaScript; Market mutations perform stricter program validation. Play requests page one with pageSize 100 and active state, then applies its own twelve-item pages. Only the first one hundred results after helper filtering and sorting can appear even though the RPC query fetched all matching accounts. Orders filters that same retained active page by wallet, so it is neither complete seller history nor a view of persistent sold accounts.

Refresh has an additional startup edge. lastChainRefreshAt begins at zero, openPanel starts a quiet non-forced refresh, and the cooldown compares that zero with performance.now. Opening Market during roughly the first twelve seconds of a page session can therefore return cooldown without making the first RPC query or displaying an error. Later category, search, currency, and sort interactions rerender locally; only a manual or allowed refresh changes the retained one-hundred-result set.

Chain categories and item presentation collapse during normalization. Any item slot becomes equipment and any block slot becomes raw. For an item, Play keeps only a generic tool kind, numeric item ID, quantity, durability, maximum durability, and quality; it drops itemCode, item PDA, volume, grade, level, metadata, material identity, and forged design data. Chunk.js consequently chooses the basic pickaxe model even for a smelted material or distinct forged design, and the name often becomes a numeric item ID. The on-chain Listing's sourceSlot still drives settlement, while rawListing preserves a decoded copy for inspection, so this is a merchandising and reconstruction defect rather than a slot-settlement substitution. Source type survives as backpack or equipment, but the current sell flow always submits backpack, and building or clothing never survives a chain round trip.

Local listings are a different and currently mostly unreachable system. The branch would remove a complete object from local game state, serialize its snapshot under nicechunk.play.marketListings.v1, and display an eight-hex-digit 32-bit browser hash. Current PDA-only GameState makes consumeBackpackItems fail and addBackpackSlotSnapshot return null, so new local creation cannot normally complete and an old stored order cannot normally be returned through that path. Previously stored entries may still load and display, but no Listing PDA, rent, program escrow, token transfer, or chain settlement exists.

Samples are a third state. Whenever both in-memory chain and active-local arrays are empty, Browse substitutes four hardcoded preview cards; that can follow a confirmed zero-result query, but it can also follow the startup cooldown, an unavailable module, or an RPC failure. Seeing samples alone therefore does not prove that a live query succeeded. The Buy handler cannot find a sample in state.chainListings and warns that samples are previews only. Dated mutable audit note: on 2026-07-15 at confirmed slot 476,305,674, Devnet returned zero 216-byte accounts for the unified Game Program query; later rechecks through confirmed slot 476,310,786 again found zero. The retired standalone Market address still owned seven 216-byte version-3 Listing accounts, but the current client never scans that program, so they were not active website liquidity. With no local order, the current panel therefore showed all four samples at that snapshot. Future listings can make this statement stale.

The Market panel also bypasses the project's locale dictionaries today. Its visible HTML labels, aria labels, runtime statuses, item names, and pager copy are hardcoded in English instead of using data-i18n attributes or t(). The current Play i18n checker collects existing translation references but does not flag these unmarked strings, so a green check does not mean this panel is localized.

VERIFY BYTES, THEN RECONSTRUCT PIXELS

A verifier should follow the Listing, balances, and Backpack record instead of trusting the card

For creation, verify the transaction targeted Game namespace four, the Listing address matches seller and ID, the account has magic NCKMKT01 and version five, state is active, price and currency are intended, and sourceSlot equals the record removed from the seller Backpack. That proves a complete record entered program custody. It does not prove a human category, fair price, off-chain description, visual model, or provenance beyond the fields already inside that slot.

For purchase, verify the buyer differs from seller, the pre-purchase Listing was active, the fixed mint or treasury route was used, seller and treasury balance changes match integer fee math, the buyer Backpack gained the exact stored record, and the retained Listing names the buyer and sale's Solana Clock slot. A capacity precheck, wallet popup, transaction signature string, or optimistic status message is insufficient without confirmed account state.

Cancellation needs different evidence because its account disappears. Verify the successful transaction accounts, the seller-owned destination Backpack and appended record, that the Listing's pre-transaction lamport balance was credited to the seller while accounting for the network fee, and absence of the former Listing PDA. An explorer cannot later decode a persistent canceled state from zeroed data. Sold accounts are easier to inspect but also expose the missing cleanup and continuing rent cost.

Chunk.js can reconstruct forged equipment, smelting materials, resource drops, and baked block icons when it receives their compact identities and visual metadata. The current Market normalizer does not preserve enough of those fields for chain items: it sends a generic tool object and therefore selects the basic pickaxe model. Rich Backpack reconstruction elsewhere in Play does not repair this separate Market card path. In either case, geometry, cyan color, category copy, and artwork composition are not fields in ListingAccount. The guide illustration is an accessible process narrative, not a claim that the browser captured an on-chain model. Current tests cover Market state packing and fee helpers but do not program-test create, buy, or cancel and do not behavior-test the Play panel, so these lifecycle claims come from source-path and live-account audit rather than an end-to-end market test. The active Devnet unified program is upgradeable, and this review did not establish byte-for-byte parity between its deployed binary and the working-tree source.

Purchase proof Balances, buyer Backpack, sold metadata

All three outcomes must agree with the same confirmed instruction.

Visual meaning Client reconstruction

Chunk.js geometry and labels are derived from slot identity, not stored in the Listing account.

What the code guarantees Follow the same eighty bytes across custody

The strongest market verification path compares seller removal, Listing sourceSlot, and buyer append. Names, thumbnails, categories, and local hashes can help a player navigate, but they are not substitutes for that record-level transition.

ADDRESS, ESCROW, PAYMENT, AND LIFECYCLE EQUATIONS

Five equations separate the committed trade from its browser presentation

The address and layout equations identify the active unified program and exact escrow bytes. Decimal and fee equations expose base-unit rounding and the checked-multiplication ceiling. The final transition shows why payment, delivery, cancellation, and rent behavior must be evaluated as complete atomic state changes.

Unified Market routing and deterministic authorities

NCK and SOL decimal text becomes an unsigned base-unit price

d_NCK = 6; d_SOL = 9; p = whole*10^d + integer(rightPad(fraction,d)); 1 <= p <= 2^64-1; fractionLength <= d

The chain helper accepts nonnegative decimal text, rejects more than the configured decimal places, pads the fractional part on the right, and encodes a positive u64. NCK settlement uses transfer_checked with six decimals and the fixed Devnet mint. SOL settlement transfers the stored integer as lamports; nine-decimal conversion is a client convention rather than a decimal field in ListingAccount. Current Play uses a 0.000001 input step for both currencies and converts the form value to a JavaScript Number before calling this helper, so its form does not cleanly expose nine-decimal SOL increments and scientific notation or precision loss can prevent the exact parser from receiving the user's original text.

d
Client decimal precision: six for NCK and nine for SOL.
p
Positive base-unit price committed in the Listing and later used without a buyer override.
rightPad
Zero-padding that makes a short fractional string an exact base-unit integer.

Hardcoded one-percent fee, floor rounding, and overflow ceiling

f = floor(p*100/10000); sellerAmount = p-f; require p*100 <= 2^64-1, equivalently p <= 184467440737095516; otherwise InvalidFee; f = 0 for 1 <= p <= 99

Market uses its own compile-time 100 basis-point constant and checked u64 multiplication before division. The current Core global-config value also happens to be 100, but the Market instruction never reads that account, so changing Core configuration would not change settlement. Creation accepts any positive u64, while a later purchase of a price above the multiplication ceiling fails. Integer floor rounding also means sub-one-fee-base-unit trades send nothing to treasury.

Fee in the same base units as the stored price.

Atomic purchase, capacity, cancellation, and rent outcomes

market balances only: buy: (F_b,F_s,F_t,B_b,A_active,r) -> (F_b-p,F_s+sellerAmount,F_t+f,B_b||escrowSlot,A_sold,r), require buyer != seller and |B_b| < capacity; cancel: (B_s,A_active,r) -> (B_s||escrowSlot,closed,0) and sellerLamports += r; failure => Market payment transfers and program-account mutations unchanged, but network fee may remain charged; A_sold has no close path

Excluding network fees and any auxiliary token-account rent, a successful SOL or NCK purchase pays the seller, pays treasury only when the rounded fee is nonzero, strictly appends the stored record to a buyer-owned Backpack, and writes buyer and sold metadata. Solana transaction atomicity reverses those Market transfers and program-account changes if Backpack validation, capacity, append, or sold marking fails, while the fee payer can still be charged for processing the failed transaction. Cancellation instead appends to a seller-owned Backpack, zeros Listing data, drains all Listing lamports to the seller, and closes the account. A sold Listing remains allocated with its rent, cannot pass the active-only cancellation check, and has no enabled cleanup instruction.

B_b, B_s

A_active, A_sold, r

The fee is a checked hardcoded integer split

Rust programs/nicechunk_market/src/lib.rs
fn split_market_payment(price_base_units: u64) -> Result<(u64, u64), NicechunkMarketError> {
    let fee = price_base_units
        .checked_mul(MARKET_FEE_BPS as u64)
        .ok_or(NicechunkMarketError::InvalidFee)?
        / BPS_DENOMINATOR;
    let seller_amount = price_base_units
        .checked_sub(fee)
        .ok_or(NicechunkMarketError::InvalidFee)?;
    Ok((seller_amount, fee))
}

MARKET_FEE_BPS is a private constant set to 100 in this program. Multiplication is checked before division, rounding is downward in base units, and no GlobalConfig account is present in the purchase account list.

    **seller.try_borrow_mut_lamports()? = seller
        .lamports()
        .checked_add(listing_lamports)
        .ok_or(NicechunkMarketError::InvalidListingData)?;
    **listing.try_borrow_mut_lamports()? = 0;

Although the processor's initial guard accepts account lists of length two, three, or five, the active return path subsequently requires exactly five accounts. The canceled byte is immediately erased by data.fill(0), then every lamport is transferred to the seller, so a successful cancellation leaves no durable canceled Listing account.

cancelMarketListingOnChain can accept sourceInventory and only adds the Backpack, Backpack Program, and market-authority keys when that value exists. This call passes neither sourceInventory nor another recoverable address. It therefore builds only seller and Listing keys, and the program rejects the transaction with InvalidAccountCount before returning escrow.
