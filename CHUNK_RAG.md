# Nicechunk Chunk / Guardian Region RAG

## Goal

Update the current Nicechunk chunk and watcher design to support a fair, decentralized, Solana/MagicBlock-friendly blockchain game model.

The design principle is:

```text
Procedural terrain is generated by deterministic seed/config.
Do not store full terrain on chain.
Only store persistent player-made deltas and important settlement state.
Guardian / Watcher regions should be large enough to support real gameplay, not just a tiny technical demo.
```

## Core Chunk Model

The world uses a chunk size of:

```text
chunkSize = 16
```

A chunk covers the X/Z plane:

```text
localX: 0..15
localZ: 0..15

worldX = chunkX * 16 + localX
worldZ = chunkZ * 16 + localZ
```

Each chunk contains:

```text
16 x 16 = 256 terrain columns
```

Chunk height is not fixed to 16. Terrain height is generated procedurally per X/Z column.

However, blockchain-authoritative operations must have vertical limits.

Add these global world bounds:

```text
minBuildY = -32
maxBuildY = 256
maxTerrainHeight = 160
seaLevel = 48
sectionHeight = 16
```

Recommended first version:

```json
{
  "chunkSize": 16,
  "sectionHeight": 16,
  "minBuildY": -32,
  "maxBuildY": 256,
  "maxTerrainHeight": 160,
  "seaLevel": 48
}
```

Reason:

```text
The browser can procedurally generate variable terrain height.
But chain-side validation must reject invalid or abusive Y coordinates.
Players must not be able to create infinite placed-block or removed-block state at arbitrary heights.
```

## Coordinates And Client Keys

Client-side keys can remain as strings:

```text
Block key: "x,y,z"
Chunk key: "chunkX,chunkZ"
```

But chain-side state must not use string keys.

Chain-side block coordinates should use compact binary values:

```rust
chunk_x: i32
chunk_z: i32
local_x: u8      // 0..15
local_z: u8      // 0..15
y: i16
block_type: u16
```

Chunk PDA seed example:

```text
["chunk", world_id, chunk_x_le_bytes, chunk_z_le_bytes]
```

Do not create one PDA per block. That would be too expensive and cause account explosion.

## Persistent Runtime State

Current browser state:

```js
{
  solidBlocks: Set,
  generatedSolidRefs: Map,
  removedBlocks: Set,
  placedBlocks: Map,
  dynamicWater: Set,
  blockDamage: Map
}
```

For chain or MagicBlock settlement, only these are persistent:

```text
removedBlocks
placedBlocks
```

Optional later:

```text
water source changes
important asset-related state
```

Do not permanently settle:

```text
blockDamage
temporary mining progress
pure rendering/cache state
```

`blockDamage` is temporary interaction state and should stay client-side or MagicBlock/local ephemeral state unless the game explicitly needs partially damaged blocks to persist.

## Dynamic Water Policy

For MVP, dynamic water should not be chain-authoritative.

Do not store every dynamic water cell on chain.

Reason:

```text
Water flow creates chain reactions.
One mined block may affect many cells and multiple chunks.
Players could abuse water to create excessive settlement state.
It increases write cost and complexity.
```

MVP rule:

```text
Source water is generated from deterministic terrain.
Dynamic water is visual/client-side or ephemeral.
Only source-water placed/removed events may become persistent later if needed.
```

If water must become authoritative later, store only compact source changes:

```text
waterSourceRemoved
waterSourcePlaced
```

Do not store full flowing water grids.

## Chunk Delta Storage

Do not store `removedBlocks` as an infinite Set on chain.

Use vertical sections.

A section is:

```text
16 x 16 x 16 blocks
```

This is:

```text
4096 blocks
```

Removed blocks can be represented as a bitmap:

```text
4096 bits = 512 bytes
```

Recommended structure:

```rust
pub struct ChunkAccount {
    pub world_id: u16,
    pub chunk_x: i32,
    pub chunk_z: i32,

    pub guardian_region: Pubkey,

    pub version: u32,
    pub delta_root: [u8; 32],
    pub state_hash: [u8; 32],

    pub mined_count: u32,
    pub placed_count: u32,

    pub last_commit_slot: u64,
    pub dirty_seq: u64,

    pub bump: u8,
}
```

Section delta:

```rust
pub struct ChunkSectionDelta {
    pub chunk: Pubkey,

    // y_band = floor(y / 16)
    pub y_band: i16,

    // 16 * 16 * 16 removed bitmap
    pub removed_bitmap: [u8; 512],

    // Placed blocks are sparse entries.
    pub placed_count: u16,
    pub placed_entries: Vec<PlacedBlockEntry>,

    pub version: u32,
    pub bump: u8,
}
```

Placed block entry:

```rust
pub struct PlacedBlockEntry {
    pub local_x: u8,
    pub local_y: u8,
    pub local_z: u8,
    pub block_type: u16,
}
```

Where:

```text
local_y = y inside the current y_band
range: 0..15
```

This is much cheaper than storing string block keys.

## Guardian / Watcher Region Model

Do not use "one watcher manages 10 chunks" as the real game setting.

10 chunks is only acceptable as a small local test/demo value.

Reason:

```text
Current renderDistance = 6
Visible chunk square = 13 x 13 = 169 chunks

Current preloadDistance = 9
Preload chunk square = 19 x 19 = 361 chunks

A watcher managing only 10 chunks cannot even cover one player's normal visible/preloaded area.
```

The design must separate three concepts:

```text
1. Client render range
2. Watcher real-time service range
3. Guardian economic/settlement region
```

## Client Rendering Range

Keep current client values:

```text
renderDistance = 6
visible chunks = (6 * 2 + 1)^2 = 169 chunks

preloadDistance = 9
preloaded chunks = (9 * 2 + 1)^2 = 361 chunks
```

This is only a rendering/performance concern.

## Watcher Real-Time Service Range

The watcher does not need to process the full economic region in real time.

It should mainly serve:

```text
player position sync
nearby player broadcast
chat
temporary local state
nearby dirty chunk delta
MagicBlock / ER state
```

Recommended value:

```text
guardianRealtimeRadiusChunks = 16
```

This covers:

```text
33 x 33 = 1089 chunks
```

This is larger than the client preload area and gives players enough room to move without constantly switching regions.

## Guardian Economic Region

The guardian economic region is the area where:

```text
resources can be activated
mining can be settled
guardian can receive tax/reward
region-level settlement can happen
```

Recommended MVP/default value:

```text
guardianRegionSizeChunks = 64
```

This means one guardian covers:

```text
64 x 64 = 4096 chunks
```

Since each chunk is `16 x 16` columns:

```text
4096 chunks * 16 * 16 = 1,048,576 terrain columns
```

In block-plane size:

```text
64 chunks * 16 blocks = 1024 blocks
```

So one guardian covers:

```text
1024 x 1024 blocks on the X/Z plane
```

This is a reasonable default for Nicechunk.

## Region Size Tiers

Recommended tiers:

```text
MVP test guardian:
32 x 32 chunks = 1024 chunks

Standard guardian:
64 x 64 chunks = 4096 chunks

Large / upgraded guardian:
100 x 100 chunks = 10000 chunks
```

Do not use:

```text
10 chunks as production guardian size
```

Also avoid ambiguous wording like:

```text
guardian radius = 100 chunks
```

Because radius 100 actually means:

```text
201 x 201 = 40,401 chunks
```

That is too large for the first version.

Use explicit square size fields instead:

```rust
pub guardian_region_size_chunks: u16;      // recommended: 64
pub guardian_realtime_radius_chunks: u16;  // recommended: 16
```

## Guardian Region Account

A guardian should not pre-create thousands of chunk accounts.

Instead, the guardian owns a region account.

Chunk accounts are created lazily only when players actually mine/place/interact in that chunk.

Recommended structure:

```rust
pub struct GuardianRegion {
    pub guardian: Pubkey,

    pub region_x: i32,
    pub region_z: i32,

    pub origin_chunk_x: i32,
    pub origin_chunk_z: i32,

    pub size_chunks: u16, // recommended: 64

    pub active: bool,

    pub total_mined_count: u64,
    pub total_tax_earned: u64,

    pub last_heartbeat_slot: u64,
    pub last_settlement_slot: u64,

    pub bump: u8,
}
```

Check whether a chunk belongs to a region:

```text
chunkX >= origin_chunk_x
chunkX <  origin_chunk_x + size_chunks
chunkZ >= origin_chunk_z
chunkZ <  origin_chunk_z + size_chunks
```

Do not store a list of all chunks inside the guardian account.

## Resource Generation Fairness

Terrain may be predictable.

Resources should not be fully predictable before mining.

Do not generate valuable resources only from:

```text
world_seed + chunkX + chunkZ + block_position
```

Because players can scan the map offline and know where every valuable resource is.

Preferred principle:

```text
Terrain can be deterministic.
Valuable resources should be revealed or finalized during mining/settlement.
```

Resource generation should include at least:

```text
world_seed_hash
chunk_x
chunk_z
local_x
y
local_z
chunk_state_hash
player_nonce
recent chain randomness or commit-reveal result
```

MVP acceptable design:

```text
chunk_state_hash + player_nonce + recent slot-based entropy
```

Formal/fair version:

```text
commit-reveal
or VRF/oracle randomness
or MagicBlock settlement randomness with anti-preview guarantees
```

Important rule:

```text
Players should not be able to know all underground rewards before actually mining.
```

## Region / Chunk / Player State Shape

Recommended high-level model:

```text
GuardianRegion
  owns one square economic region
  default size: 64 x 64 chunks
  does not pre-create all chunk accounts

ChunkAccount
  one account per active/touched chunk
  stores metadata, hashes, counters, settlement state
  does not store full terrain

ChunkSectionDelta
  one account per touched chunk section / y-band
  stores removed bitmap and sparse placed entries

PlayerRegionState
  player
  current region
  current chunk
  position
  yaw / rotation
  last chat message
  last message timestamp
  sequence number
```

Player movement and chat can remain MagicBlock/local/ephemeral unless they need Solana finality.

Do not write every movement frame to Solana mainnet.

## Commit Policy

Chunk/region state should not commit every visual action to L1 immediately.

Commit should depend on:

```text
dirty delta count
elapsed time
fee pool
important asset events
resource discovery
region handoff
chunk sleep/unload
guardian settlement interval
```

Suggested policy:

```text
Small visual/local changes stay ephemeral.
Mining that creates valuable resources settles.
Placed/removed block batches commit by section.
Guardian tax and resource rewards settle periodically.
```

Use these values for the first serious Nicechunk version:

```json
{

  "sectionHeight": 16,

  "renderDistance": 6,
  "preloadDistance": 9,

  "guardianRegionSizeChunks": 64,
  "guardianRealtimeRadiusChunks": 16,

  "minBuildY": -32,
  "maxBuildY": 256,
  "maxTerrainHeight": 160,
  "seaLevel": 48
}
```

Summary:

```text
Chunk size remains 16 x 16.
The chain stores only deltas, not generated terrain.
A standard guardian covers 64 x 64 chunks.
The watcher real-time service radius is 16 chunks.
Dynamic water is not chain-authoritative in MVP.
Removed blocks use section bitmaps.
Placed blocks use sparse typed entries.
Resources should not be fully predictable before mining.
```

## Compressed Changed Block State Plan

Decision recorded for the next `nicechunk_chunk` redesign:

```text
Do not store full terrain on-chain.
Do not store full per-change history in the chunk PDA.
Store only the current changed block state for blocks that differ from deterministic terrain.
```

The current devnet chunk program uses a larger audit-style delta entry. The future production-oriented design should replace that with a compact current-state encoding.

Recommended binary entry:

```text
5 bytes per changed block

bits 0..3    localX    // 0..15
bits 4..7    localZ    // 0..15
bits 8..16   yOffset   // y - minBuildY, enough for -32..256
bits 17..32  blockId   // 0 = air/removed, >0 = placed/replaced block type
```

Meaning:

```text
blockId = 0
  The generated or placed block at this coordinate is removed.

blockId > 0
  The coordinate currently resolves to this block type.
```

This chunk PDA should answer only one question:

```text
What changed blocks currently override deterministic world generation in this chunk?
```

It should not store:

```text
actor pubkey
timestamp
slot
previousBlockId
full event history
resource settlement results
```

Those belong in separate event/history/indexer/settlement systems if needed.

Recommended account size:

```text
32 KB per chunk PDA for the first serious version.
64 KB only if testing shows 32 KB is too small.
```

Approximate capacity and rent:

```text
32 KB account:
  ~6,000 compact changed-block entries
  ~0.23 SOL rent-exempt cost per chunk

64 KB account:
  ~12,000 compact changed-block entries
  ~0.46 SOL rent-exempt cost per chunk
```

Rationale:

```text
The world can be arbitrarily large, but unchanged generated terrain costs zero chain storage.
Most chunks should have few changed blocks.
Compact binary storage is much cheaper than text.
Text/base64/base58 encodings may be used for debugging and display, but not as the chain storage format.
```

Debug display may encode each 5-byte entry as a short text token, but the account data should remain binary.
